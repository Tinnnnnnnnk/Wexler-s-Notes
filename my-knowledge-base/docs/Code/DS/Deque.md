# ⚔️ Java 终极兵器：Deque (双端队列) 完全指南



> [!abstract] 核心定义
> `Deque` (发音为 "deck"，全称 **D**ouble **E**nded **Que**ue)，即**双端队列**。
> 想象它是一根**两头都开口的管子**：你可以从左边塞东西、拿东西，也可以从右边塞东西、拿东西。这种极其自由的物理特性，让它成为了 Java 集合里的“变形金刚”。

---

## 🛠️ 怎么创建它？(只认准 ArrayDeque)

`Deque` 在 Java 中只是一个**接口 (Interface)**，你不能直接 `new Deque()`。它有两个最著名的实现类：`LinkedList` 和 `ArrayDeque`。

**架构师绝对铁律：99% 的情况下，无脑使用 `ArrayDeque`！**
```java
// ✅ 满分标准写法 (底层是物理连续的动态数组，光速极快，极其省内存)
Deque<Integer> deque = new ArrayDeque<>();

// ❌ 极度不推荐 (底层是离散的节点指针，疯狂 new 对象，产生海量内存碎片)
Deque<Integer> slowDeque = new LinkedList<>(); 
```

## 🤖 变形金刚的三种终极形态 (API 兵器谱)

因为 `Deque` 两头通吃，所以它可以完美扮演**三种完全不同的角色**。你必须根据具体的算法题，选择对应的 API 连招：

### 形态一：伪装成【栈 Stack】(后进先出 LIFO)
*这是完美替代老旧 `Stack` 类的官方指定用法！*
* **物理动作：** 封死管子的右边，只在**左边（头部）**进出。
* **API 连招：**
  * `deque.push(e)`：压栈（把元素硬塞进头部）。
  * `deque.pop()`：弹栈（把头部的元素拔出来并返回）。
  * `deque.peek()`：偷看（只看看头部是谁，不拔出来）。
* **实战场景：** 括号匹配、深度优先搜索 (DFS)、单调栈运算。

### 形态二：伪装成【队列 Queue】(先进先出 FIFO)
*这是最标准的排队买票模型。*
* **物理动作：** 从**右边（尾部）**排队进去，从**左边（头部）**结算出来。
* **API 连招：**
  * `deque.offer(e)`：入队（乖乖排在尾部）。
  * `deque.poll()`：出队（头部第一个人结算走人）。
  * `deque.peek()`：偷看（看看现在排在最前面的是谁）。
* **实战场景：** 层序遍历 (树的 BFS)、广度优先搜索、消息队列。

### 形态三：【双端队列完全体】(高级极客操作)
*这是它真正的本体，两头疯狂输出！*
* **API 连招：**
  * **头部操作：** `addFirst(e)`, `removeFirst()`, `peekFirst()`
  * **尾部操作：** `addLast(e)`, `removeLast()`, `peekLast()`
* **实战场景：** 极其硬核的**“单调队列 (Monotonic Queue)”**、滑动窗口算法。

## 🚫 避坑指南：为什么大厂封杀 `java.util.Stack`？

在 Java 早期（JDK 1.0），官方提供了一个 `Stack` 类。但由于历史遗留问题，`Stack` 继承了笨重的 `Vector` 类，导致它的每一个核心方法都加了极其沉重的 `synchronized` 线程锁。
即使你在单线程下刷算法题，它也会强行进行加锁、释放锁的无用操作，**极其拖慢运行速度**。

**官方最终宣判：** Java 官方在文档中明确标注，所有需要用到栈的场景，请全面拥抱 `Deque deque = new ArrayDeque();`，它的速度是老 `Stack` 的好几倍！
