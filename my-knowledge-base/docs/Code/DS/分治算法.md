# 🧠 分治算法基础与边界划分法则 (Divide and Conquer)

> [!abstract] 核心思想
> 分治法，顾名思义：**分而治之**。
> 就是把一个复杂的大问题，拆解成两个或多个极其相似的子问题。直到子问题简单到可以直接求解，最后再把子问题的解合并起来。
> 
> 

---

## 💡 核心顿悟：为什么必须传递“边界 (范围)”？

在处理数组、字符串等线性结构时，我们经常需要对“左半部分”和“右半部分”分别进行递归操作。

> [!failure] 错误直觉：物理切割 (Physical Slicing)
> - **做法**：真正地把原数组切开，每次新建两个小数组传给子递归。
> - **致命缺陷**：极其可怕的内存浪费（空间复杂度爆炸）和频繁复制数据的性能损耗（时间复杂度极速恶化）。

> [!success] 降维打击：虚拟划界 (Virtual Boundaries)
> - **做法**：**数组永远只有最初的那一个大全集，绝对不复制！** 我们仅仅通过传递**索引边界（`start` 和 `end`）**，在逻辑上给递归分身“画地为牢”。
> - **优势**：传递两个 `int` 类型的整数，额外开销几乎为 0。空间复杂度降到最低！

## 🛠️ 万能分治区间模板 (Java)

只要题目中同时出现 **“数组/字符串”** 和 **“递归/拆分/二叉树”**，立刻默写此模板：

```java
// 核心：永远传递原数组，外加当前子任务的领地边界 [start, end]
public Node dfs(int[] arr, int start, int end) {
    
    // 1. 领地失效拦截 (Base Case)
    // 如果起点跑到了终点后面，说明这片逻辑区间已经被榨干，没有元素了
    if (start > end) {
        return null; 
    }

    // 2. 找到本层的老大 / 分割点 (Split / Mid / Root)
    // 根据具体题目而定：可能是找中间点、最大值、或者前序遍历的根节点
    int midIndex = getMidOrRootIndex(...); 
    
    // 3. 处理本层的核心逻辑 (Conquer)
    Node root = new Node(arr[midIndex]);

    // 4. 精准给左右分身划分新地盘 (Divide)
    // ⚠️ 严禁复制数组，只修改边界！
    // 左分身的领地永远是: [当前起点, 分割点 - 1]
    root.left = dfs(arr, start, midIndex - 1);
    
    // 右分身的领地永远是: [分割点 + 1, 当前终点]
    root.right = dfs(arr, midIndex + 1, end);

    // 5. 向上级汇报 (Combine)
    return root;
}
```

## 🌐 触类旁通 (同源算法)

一旦掌握了“靠边界划定势力范围”的心法，以下著名算法的底层逻辑将变得极其透明：

1. **根据前序/中序遍历构造二叉树**：找根节点，算出左子树人数，划分左右子数组的 `[start, end]`。
2. **将有序数组转换为二叉搜索树**：找最中间的元素当根，左半区间 `[start, mid - 1]` 变左子树，右半区间 `[mid + 1, end]` 变右子树。
3. **归并排序 (Merge Sort)**：找中点切分区间，递归排好左区间和右区间，最后通过双指针合并。
4. **快速排序 (Quick Sort)**：找基准数切分区间，使得左区间都比它小，右区间都比它大，然后对左右区间继续递归。
5. **二分查找 (Binary Search)**：看一眼中点，不在中点就直接舍弃一半区间，修改 `start` 或 `end` 继续查。

> [!tip] Kasumi's Memo
> **物理结构的差异决定了传参的方式：**
> - **二叉树 (`TreeNode`)**：物理上天然隔离，`root.left` 传进去绝对跑不到右边，不需要传范围。
> - **数组 (`Array`)**：所有数据挤在大平层里，必须用指针（下标 `start`, `end`）建立无形的墙！
