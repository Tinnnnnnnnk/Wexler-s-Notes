# 437. 路径总和 III (Path Sum III)

> [!info]- 题目描述 (点击展开)
> > 给定一个二叉树的根节点 `root` ，和一个整数 `targetSum` ，求该二叉树里节点值之和等于 `targetSum` 的 **路径** 的数目。
> > 
> > **注意：**
> > - 路径 **不需要** 从根节点开始，也不需要在叶子节点结束。
> > - 但是路径方向必须是向下的（只能从父节点到子节点）。

---

## 💡 解题思路 (Kasumi's Memo)

### 核心策略 (前缀和 + 哈希表回溯)
1.  **第一步（初始化账本）：** 建立一个 HashMap 记录 `[前缀和 : 出现次数]`。**极其重要**：必须先放入一个初始记录 `put(0L, 1)`，以防刚好从根节点开始的合法路径被漏掉。
2.  **第二步（计算当前状态）：** 在 DFS 遍历时，带上从上级传下来的累加和，加上当前节点的值，得到当前的绝对前缀和 `currSum`。
3.  **第三步（光速查账与登记）：** 去哈希表里查 `currSum - targetSum` 是否存在。如果存在，说明历史某个节点到当前节点刚好凑够了 target，把次数累加到结果中。然后把自己的 `currSum` 登记进哈希表。
4.  **第四步（灵魂回溯）：** 派分身去左右子树寻找。当分身都回来、准备返回上一层（下班离开该节点）时，**必须**从哈希表中把当前 `currSum` 的次数减 1，擦除脚印，防止污染另一条树枝的计算！

### 为什么选择这个解法？
- **降维打击：** 彻底抛弃了在 List 中遍历加减的 $O(N^2)$ 或 $O(N \times H)$ 的笨方法。利用哈希表 $O(1)$ 的查询特性，让时间复杂度暴降至绝对的 **$O(N)$**。

## 💻 代码实现 (Java)

```java
class Solution {
    // 账本：记录 [前缀和 : 出现次数]
    // 注意：前缀和可能会非常大，必须用 Long 类型！
    Map<Long, Integer> prefixMap = new HashMap<>();
    int totalCount = 0;

    public int pathSum(TreeNode root, int targetSum) {
        // 关键防漏：初始化“什么都不选”的前缀和为 0，出现 1 次
        prefixMap.put(0L, 1);
        dfs(root, 0L, targetSum);
        return totalCount;
    }

    private void dfs(TreeNode root, long currSum, int targetSum) {
        if (root == null) return;

        // 1. 算总账：当前路径的前缀和
        currSum += root.val;

        // 2. 查账：看看历史记录里，有没有刚好能凑出 target 的前缀和
        // 公式推导：currSum - 历史前缀和 = targetSum  =>  历史前缀和 = currSum - targetSum
        long targetPrefix = currSum - targetSum;
        totalCount += prefixMap.getOrDefault(targetPrefix, 0);

        // 3. 记账：把自己的前缀和登记入册，然后派分身去左右子树
        prefixMap.put(currSum, prefixMap.getOrDefault(currSum, 0) + 1);
        
        dfs(root.left, currSum, targetSum);
        dfs(root.right, currSum, targetSum);

        // 4. 销账 (核心回溯)：离开当前节点，必须擦除属于自己的脚印！
        prefixMap.put(currSum, prefixMap.get(currSum) - 1);
    }
}
```

## 📊 复杂度分析
- **时间复杂度：** $O(N)$
  - *分析：* 完美的一遍 DFS。每个节点只会被访问一次，哈希表的存取操作时间复杂度为 $O(1)$，因此整体时间复杂度为线性。
- **空间复杂度：** $O(N)$
  - *分析：* 空间消耗取决于哈希表的大小（最坏情况下存储 $N$ 个不同的前缀和）以及递归调用栈的深度 $H$。总空间复杂度为 $O(N)$。

## 📝 错题本 / 复盘
> [!failure] 遇到的坑 / 错误点
> > 1. **大数溢出陷阱 (Integer Overflow)：** 试图用 `int` 甚至 `List<Integer>` 来累加前缀和。测试用例中有连续十亿级别的数值，导致 `int` 溢出变成负数，彻底搞乱了匹配逻辑。**必须强制使用 `long`！**
> > 2. **遗漏初始空气账 `(0, 1)`：** 如果不把前缀和 0 放入表中，那么那些“起点刚好是根节点”的正确路径就会被完美错过。
> > 3. **忘记回溯擦除脚印：** 误以为只有 List 才需要 remove。其实只要是“全局共享状态”（如这里的 Map），在退出当前树枝时都必须恢复原状，否则左子树的累计值会错误地被右子树读取。

> [!tip] 关键技巧 / 总结
> > **公式转移法：** 找目标子路径 `[A -> B] == target`，等价于寻找 `[Root -> B] - [Root -> A的父节点] == target`。这种利用历史差值来锁定区间的方法，是所有“连续区间求和”问题的神级套路。

## 🧠 深度思考与感悟 (Reflections)
- **💡 触类旁通：** 这一套“前缀和+哈希表”完全可以直接平移到一维数组上解决大名鼎鼎的 **LeetCode 560: 和为 K 的子数组**。唯一的区别是数组不需要回溯（因为数组是一条路走到黑，没有树的“岔路口”）。
- **🔧 工程思维：** 空间换时间！我们花了一点点内存建了一个 HashMap 账本，换来的是查询时间的几何级数暴降。在服务器开发中，这种缓存历史计算结果的思维无处不在。
- **✨ 瞬间感悟：** 回溯的本质根本不是“List.remove()”，而是**“状态的隔离与重置”**。只要涉及跨分支共享的全局变量，就一定要思考：我离开的时候，是不是该把舞台恢复成我刚来时的样子？
