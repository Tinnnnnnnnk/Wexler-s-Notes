# 207. 课程表 (Course Schedule) - Kahn 算法 (拓扑排序)

> [!info]- 题目描述 (点击展开)
> > 你这个学期必须选修 `numCourses` 门课程，记为 `0` 到 `numCourses - 1`。
> > 在选修某些课程之前需要一些先修课程。 例如，想要学习课程 0 ，你需要先完成课程 1 ，我们用一个匹配来表示：`[0, 1]`。
> > 请你判断是否可能完成所有课程的学习？如果可以，返回 `true`；否则，返回 `false`。

---

## 💡 解题思路 (Kasumi's Memo)

### 核心策略 (顺藤摸瓜剥洋葱法)
彻底抛弃极其耗时的 `list.remove()` 操作，转而使用“入度数组”和“邻接表”进行 $O(1)$ 的状态更新。

1.  **第一步（建图与统计）：** 准备两个核心神器。
    * `inDegree` 数组：记录每门课**还需要几门前置课**（入度）。
    * `adjacency` 邻接表：记录这门课学完后，**能解锁哪些后续课程**。
2.  **第二步（寻找天才选修课）：** 遍历一次 `inDegree` 数组，把所有入度为 0 的课（不需要任何前置条件的起点）全部扔进队列 `Queue` 中。
3.  **第三步（BFS 顺藤摸瓜）：** 只要队列不为空，就弹出一门课（代表这门课修完了）。
    * 让已修课程的总数 `count + 1`。
    * 去邻接表里查它能解锁哪些后续课。把这些后续课的入度统统减 1 (`inDegree[next]--`)。
    * **灵魂判断：** 如果某门后续课的入度刚好减到了 0，说明它的前置条件全满足了，立刻把它也扔进队列！
4.  **第四步（终局核对）：** 队列空了之后，看看我们总共修完了几门课（`count`）。如果 `count == numCourses`，说明全部修完，没有死锁的环；否则说明图里有环，返回 `false`。

### 为什么选择这个解法？
- **极致的性能降维：** 你的原版暴力解每次删除前置条件都要用 `contains` 和 `remove(Object)`，底层是 $O(N)$ 遍历。而 Kahn 算法用数组索引 `inDegree[next]--` 进行扣减，是绝对的光速 $O(1)$！整体时间从 $O(V^2 + E)$ 暴降到了 $O(V + E)$。

## 💻 代码实现 (Java)

```java
import java.util.*;

class Solution {
    public boolean canFinish(int numCourses, int[][] prerequisites) {
        // 1. 核心神器初始化
        int[] inDegree = new int[numCourses]; // 记录入度（还需要几门前置课）
        List<List<Integer>> adjacency = new ArrayList<>(); // 邻接表（我能解锁谁）
        
        for (int i = 0; i < numCourses; i++) {
            adjacency.add(new ArrayList<>());
        }
        
        // 2. 填写账本（极其关键：注意谁是指向谁的！）
        // 规则是 [0, 1] 代表想学 0，先学 1。
        // 所以 1 是前置，0 是后续。方向是 1 -> 0
        for (int[] info : prerequisites) {
            int course = info[0]; // 要学的课
            int pre = info[1];    // 前置课
            
            inDegree[course]++; // 要学的课，前置条件数量 + 1
            adjacency.get(pre).add(course); // 前置课学完，能解锁这门要学的课
        }
        
        // 3. 寻找全部起点（不需要前置的课）排队入场
        Queue<Integer> queue = new LinkedList<>();
        for (int i = 0; i < numCourses; i++) {
            if (inDegree[i] == 0) {
                queue.offer(i);
            }
        }
        
        // 4. BFS 顺藤摸瓜剥洋葱
        int count = 0; // 记录我们到底修了几门课
        
        while (!queue.isEmpty()) {
            int cur = queue.poll(); // 取出一门可以修的课
            count++; // 修完啦！
            
            // 去查这门课能解锁哪些后续课程
            for (int nextCourse : adjacency.get(cur)) {
                inDegree[nextCourse]--; // 后续课程的前置条件减 1
                
                // 如果前置条件全清空了，说明它可以修了，赶紧排队！
                if (inDegree[nextCourse] == 0) {
                    queue.offer(nextCourse);
                }
            }
        }
        
        // 5. 终局核对：修完的课是不是等于总课数？
        return count == numCourses;
    }
}
```

## 📊 复杂度分析
- **时间复杂度：** $O(V + E)$
  - *分析：* $V$ 是节点数（课程数），$E$ 是边数（先修要求数）。建图过程需要遍历所有的边 $O(E)$。在 BFS 过程中，每个节点最多入队出队一次 $O(V)$，每条边最多被访问一次 $O(E)$。总时间极其高效。
- **空间复杂度：** $O(V + E)$
  - *分析：* 邻接表需要存储所有的边 $O(E)$，入度数组和队列的长度最大为节点数 $O(V)$。

## 📝 错题本 / 复盘
> [!failure] 遇到的坑 / 错误点
> > 1. **方向搞反的灾难：** 题目里的 `[0, 1]` 是“先修 1，再修 0”，所以在图里的边应该是 `1 -> 0`。在构建邻接表时，绝不能写反了，否则 BFS 的顺藤摸瓜就会顺着反方向摸，直接全盘崩溃。
> > 2. **迷信 `Map` 和 `List` 的动态删除：** 频繁使用 `remove(Object)` 会导致性能急剧下降，并且极易踩中 Java 基础类型装箱的坑（误调 `remove(int index)`）。

> [!tip] 关键技巧 / 总结
> > **入度计数的艺术：** 当你需要频繁判断一个集合是否为空时，不妨退一步，看看能不能**只维护这个集合的 `size`**（也就是入度数字）！很多时候我们根本不关心前置条件具体是哪些课，只关心“数量是不是变成了 0”。
