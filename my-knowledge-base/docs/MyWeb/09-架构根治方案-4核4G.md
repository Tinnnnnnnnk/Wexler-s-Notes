# 架构根治方案（4核4G可落地）

## 1. 结论先说

可以完成你的需求，而且不需要上重型后端。

根治方向不是“继续堆特效”，而是把现有站点改造成：

1. 文档内核层（VitePress，负责 md 内容渲染）
2. 首页交互层（Home Lab，负责复杂视觉与交互）
3. UI 系统层（统一状态、层级、动画、性能降级）

这套方案能在 4核4G 服务器稳定运行，因为最终仍然是静态站点 + Nginx，服务器不做重计算。

---

## 2. 当前根因（为什么会“越改越乱”）

你现在的痛点不是单一 Bug，而是“体系冲突”：

1. 全局类名驱动过多：多个组件都在改 `document.documentElement.classList`。
2. 样式文件过大且历史规则叠加：新旧选择器互相覆盖，层级难推断。
3. 动画系统混用：`reveal-*`、模式切换动画、玻璃/液态动画并行，状态边界不清晰。
4. 层级协议缺失：`z-index` 分散定义，容易出现“内容存在但被覆盖”。
5. 首页与文档页耦合：本应只影响首页的样式影响到全站。

---

## 3. 目标架构（不换 VitePress 的前提下）

### 3.1 模块边界

1. `docs/.vitepress/theme/components/core/*`
- 全局基础能力：命令面板、阅读增强等。

2. `docs/.vitepress/theme/components/home-lab/*`
- 只给首页使用：布局场景、风格切换、背景层、播放器。

3. `docs/.vitepress/theme/components/editor/*`
- 编辑器能力：画布、状态、导入导出。

4. `docs/.vitepress/theme/stores/*`
- 统一状态源（home fx、home layout、editor 状态分离）。

5. `docs/.vitepress/theme/styles/*`
- 分层样式，不再只靠一个超长 `style.css`。

### 3.2 推荐目录

```text
my-knowledge-base/docs/.vitepress/theme/
├─ index.js
├─ stores/
│  ├─ uiModeStore.js
│  ├─ homeLayoutStore.js
│  └─ editorStore.js
├─ components/
│  ├─ core/
│  ├─ home-lab/
│  │  ├─ HomeLayoutToggle.vue
│  │  ├─ HomeLayoutScenes.vue
│  │  ├─ HomeFxBackdrop.vue
│  │  └─ LiquidPlayer.vue
│  └─ editor/
├─ styles/
│  ├─ tokens.css
│  ├─ layers.css
│  ├─ base.css
│  ├─ home-lab.css
│  ├─ effects.css
│  └─ editor.css
└─ style.css  (仅聚合 import，不再堆业务规则)
```

## 4. 统一规范（从根源防屎山）

### 4.1 状态规范

只保留一个 UI 模式入口（单向数据流）：

1. `mode.fx`: `default | glass | liquid`
2. `mode.layout`: `minimal | dashboard | editorial`
3. `mode.performance`: `normal | safe`

禁止多个组件直接改 html 全局类。

### 4.2 层级规范（必须固定）

```text
--z-bg: 0               背景层
--z-content: 10         页面内容主层
--z-nav: 40             顶部导航
--z-overlay: 60         功能浮层
--z-editor: 80          编辑器
--z-modal: 100          弹层/遮罩
```

任何新组件必须引用 token，不得随意写魔法数字。

### 4.3 动画规范

1. 大面积动效只允许 `transform/opacity`。
2. `backdrop-filter` 限制在核心卡片，不做全屏级频繁变化。
3. 检测低性能时自动进入 `safe` 模式（降低 blur、关闭 blob 动画）。
4. 统一动效时长：`120ms / 240ms / 420ms` 三档。

### 4.4 样式隔离

1. 首页样式统一挂在 `.home-lab` 命名空间。
2. 文档页样式统一挂在 `.VPDoc` 命名空间。
3. 禁止裸选择器直接改 `.Layout`, `.VPHome`, `.VPContent`（除基础层）。

## 5. 分阶段实施计划（可直接执行）

## Phase 0：基线冻结（0.5 天）

1. 给当前可用版本打 tag：`baseline-home-lab`。
2. 导出当前 `style.css`、`theme/index.js`、核心组件备份。
3. 建立 `smoke` 检查清单（首页/文档页/移动端/三模式切换）。

验收：能一键回退到基线版本。

## Phase 1：目录与状态重构（1 天）

1. 建立 `stores/uiModeStore.js`，接管原多处模式逻辑。
2. 把首页组件移动到 `components/home-lab/`。
3. `theme/index.js` 只负责装配，不承载业务逻辑。

验收：三模式切换可用；不存在双写 class 的逻辑。

## Phase 2：样式分层重构（1~1.5 天）

1. 抽 `tokens.css`（颜色/圆角/z-index/动效变量）。
2. 抽 `home-lab.css`（首页专属）。
3. 抽 `editor.css`（编辑器专属）。
4. `style.css` 只做 import：

```css
@import './styles/tokens.css';
@import './styles/layers.css';
@import './styles/base.css';
@import './styles/home-lab.css';
@import './styles/effects.css';
@import './styles/editor.css';
```

验收：删除旧大段规则后页面行为不变。

## Phase 3：层级冲突治理（0.5~1 天）

1. 全量替换散落 `z-index` 为 token。
2. 去掉首页与文档页互相影响规则。
3. 增加“层级断言样式”（调试开关）。

验收：不存在“可复制但看不见”面板。

## Phase 4：性能治理（1 天）

1. 首页三模式都加 `performance safe` 降级策略。
2. 滚动、拖拽、resize 全部 `requestAnimationFrame` 节流。
3. 避免多处 `MutationObserver` 重复监听 html class。

验收：模式切换与滚动不卡顿，CPU 峰值明显下降。

## Phase 5：发布与回滚体系（0.5 天）

1. CI 加两步：`docs:build` + 关键页面快照检查。
2. 生产部署保留上一个版本目录（秒级回滚）。
3. 记录 `CHANGELOG-UI.md`。

验收：发布失败可自动回退。

## 6. 4核4G 资源预算

## 6.1 生产运行（静态站点）

1. Nginx：几十 MB ~ 200MB（常见）
2. 静态资源：磁盘占用为主，内存压力低
3. 并发瓶颈主要来自带宽，不是 CPU/内存

结论：4核4G 充足。

## 6.2 构建策略

1. 构建放 GitHub Actions（推荐）
2. 服务器只接收构建产物（rsync）

结论：服务器负担最小。

## 6.3 若启用在线编辑后端

推荐轻量方案：

1. `Fastify + SQLite` 单容器
2. 不上 Redis/ES
3. 加一个简单鉴权（JWT 或 Basic Auth）

4核4G 仍可承载个人站点规模。

---

## 7. 风险与规避

1. 风险：重构期间样式回归
- 规避：每 phase 都做页面快照 + smoke 测试

2. 风险：模式状态串线
- 规避：只有 `uiModeStore` 可写模式

3. 风险：历史 CSS 残留
- 规避：建立“废弃规则清单”，逐项删除并验证

4. 风险：上线后缓存导致“看起来没改”
- 规避：静态资源 hash + 强缓存策略 + 版本号提示

---

## 8. 验收标准（完成即算根治成功）

1. 三种布局切换后不闪烁、不消失。
2. 三种特效切换后无叠层遮挡。
3. 首页样式改动不影响文档页。
4. 文档页增强不影响首页布局。
5. 代码结构可定位：任何一个问题 5 分钟内能找到责任模块。
