## 1. Token的获取 (Login)

剧情：你拿着账号密码去换了一张 VIP 卡（Token）。

- 你的动作：POST /login (admin/admin123)
    
- 后端内部发生了什么？
    

1. UserController (前台)：接收你的账号密码。
    
2. UserService (保安)：去数据库查这个账号存不存在？密码对不对？
    
3. JwtUtils (制卡机)：验证通过后，它用加密算法生成一串字符（Token）。这串字符里藏着你的 userId 和 username。
    
4. 返回：把 Token 给你的 Apifox。
    

- 关键点：从此以后，你不用再发账号密码，只要亮出这张 Token，后端就知道“哦，是 admin 来了”。
    

## 2. Chunk Upload

剧情：你把一个大文件切成小块（虽然这次只有一块），一片片递给后端。

- 你的动作：POST /upload/chunk (带 Token, Header, File, Params)
    
- 后端内部发生了什么？ (这是最繁忙的一步)
    

1. JwtAuthenticationFilter (安检门)：
    

- 在你见到 Controller 之前，这个过滤器先拦住你：“Token 呢？”
    
- 它解析你的 Token，把里面的 userId 提取出来，放进请求的“口袋”里 (request.setAttribute)。
    

2. UploadController (包工头)：
    

- 接收所有参数 (chunkIndex, md5, file 等)。
    
- 第一道防线：如果是第 0 片，它会叫来 FileTypeValidationService，检查你的文件名（比如 .txt）是不是允许上传的类型。
    
- 获取身份：从“口袋”里掏出 userId。
    

3. UploadService (搬砖工人)：
    

- 它干了两件事：
    

- 存文件：调用 MinIO 的客户端，把这一小片文件存到 MinIO 服务器的 paismart 桶里的一个临时目录下。
    
- 记账 (关键！)：调用 FileUploadRepository (数据库)，在 MySQL 里记下一笔：“my_test_file_001 这个文件，第 0 片已经到了，总共需要 1 片”。
    
- (注：你之前报错说“文件记录不存在”，就是因为这一步的记账信息和后面合并时对不上！)
    

## 3. 合体与发令 (Merge & Kafka)

剧情：你通知后端：“砖搬完了，把它们砌成墙，然后通知 AI 团队来干活！”

- 你的动作：POST /upload/merge (带 Token, MD5, FileName)
    
- 后端内部发生了什么？ (这是逻辑最严密的一步)
    

1. UploadController (指挥官)：
    

- 查户口：调用 fileUploadRepository，问数据库：“这个 MD5 的文件，是谁传的？”
    
- 权限校验：对比数据库里的 userId 和你 Token 里的 userId。如果不对，直接报警 (403 Forbidden)。
    
- 完整性校验：问 Service：“这个文件总共几片？现在收到了几片？” (你之前报错 400 就是死在这里，系统发现 1 < 2)。
    

2. UploadService (组装大师)：
    

- 校验通过后，它去 MinIO 把所有碎片下载下来，拼成一个完整的文件。
    
- 把完整文件重新上传到 MinIO 的正式目录。
    
- 生成一个最终的文件访问链接 (URL)。
    

3. KafkaTemplate (通信兵)：
    

- 这是 Day 3 的高潮！
    
- Controller 既然已经拿到了完整的 URL，它就把这个任务打包成一个 FileProcessingTask 对象。
    
- 它大喊一声（发送消息）：“喂！Topic file-processing，来活了！这是文件的 URL，快去处理！”
    

## 4.幕后英雄 (Consumer & Vectorization)

剧情：你在前端（Apifox）收到“成功”的时候，其实好戏才刚刚开始。

- 你的动作：看着 Apifox 转圈结束，显示 200 OK。
    
- 后端内部（你看不到的地方）发生了什么？
    

1. FileProcessingConsumer (潜伏者)：
    

- 它一直蹲在 Kafka 的 file-processing 频道里监听。
    
- Controller 刚把消息发出来，它立马就收到了。
    

2. RAG 流程启动：
    

- 它根据 URL 从 MinIO 下载刚才那个完整文件。
    
- 解析：把 .txt 读成文字。
    
- 切分：把文字切成一段段。
    
- 向量化：调用 AI 模型，把文字变成数字向量。
    
- 入库：把向量存进向量数据库 (Elasticsearch/Milvus)。
