> **“去魅”**。
> 目前这个项目对你来说是一个运转中的“黑盒子”，你知道输入什么、输出什么，但不知道中间发生了什么。

> **验收标准**：随便指着架构图上的一个箭头（比如 Kafka 到 ES），你能流利地说出数据是以什么格式流动的，以及为什么要这样流。

## 🏔️ 核心高地一：基础设施与配置审计 (Infrastructure Audit)

这部分是面试的“前菜”，面试官会通过这些问题判断你对技术选型的理解深度。不要只看表面，要看懂“为什么”。

- [ ] **依赖关系深度扫描 (`pom.xml`)**
    - **LangChain4j**：项目用了哪个版本？它集成了哪些大模型的 SDK？(OpenAI? Ollama?)。
    - **Apache Tika**：重点关注 `tika-parsers`。去查一下，它默认支持哪些格式？是否引入了 `tesseract` (OCR) 相关的依赖？(这决定了你的系统能不能读扫描版 PDF)。
    - **Elasticsearch Client**：是用老旧的 `RestHighLevelClient` 还是新的 `ElasticsearchClient`？这在写查询语句时完全不同。

- [ ] **配置文件逻辑解构 (`application.yml`)**
    - **Kafka 生产参数**：重点看 `acks`、`retries`、`batch-size`。
        - *思考任务*：目前的配置是追求“高吞吐”还是“不丢失数据”？如果不记得这些参数的含义，去搜，然后标注在旁边。
    - **Redis 数据结构预判**：看配置，结合业务猜测。
        - 用了 `String` 做缓存吗？
        - 用了 `Bitmap` 吗？(如果有分片上传)。
        - 用了 `ZSet` 吗？(如果有排行榜)。
    - **线程池配置**：项目里有没有配置自定义的 `ThreadPoolTaskExecutor`？如果有，核心线程数是多少？(这关系到异步处理的并发能力)。

## 🏔️ 核心高地二：异步文件处理流水线 (The Async Pipeline)

这是你简历上**最有含金量**的模块（Kafka 解耦 + Tika 解析）。你必须能在大脑里重现一个 PDF 被“肢解”的全过程。

- [ ] **追踪目标：一个 10MB 的 PDF 文件**
    - **阶段 1：主线程的逃离 (Controller -> Producer)**
        - **核心任务**：找到 `Controller` 接收文件后，是如何把任务“甩”给 Kafka 的。
        - *深度挖掘*：发给 Kafka 的消息体 (Message Payload) 究竟长什么样？
            - 是 `{ "fileId": "123", "path": "/minio/..." }` 这种轻量级引用？
            - 还是直接把文件流塞进去了？(如果是后者，这是设计缺陷，要记下来)。
    - **阶段 2：消费者的吞吐 (Consumer -> Tika)**
        - **核心任务**：找到 `@KafkaListener`。观察这里是否开启了**多线程消费**？
        - *深度挖掘*：`Tika` 解析文件时，是把整个文件加载进内存，还是用流式读取？(去 Tika 的调用代码里看)。如果是大文件，会不会 OOM (内存溢出)？
    - **阶段 3：向量化的变身 (Text -> Vector)**
        - **核心任务**：找到 Embedding 模型的调用点。
        - *深度挖掘*：一段 500 字的文本，变成了一个多长的数组？(比如 1024 维)。这个数组最后是怎么存进 ES 的？(查看 ES 的 `Mapping` 定义，看 `dense_vector` 字段)。

## 🏔️ 核心高地三：RAG 混合检索逻辑 (Hybrid Search Logic)

这是区分“玩具 Demo”和“企业级应用”的分水岭。面试官会问你：“为什么不用纯向量检索？”

- [ ] **追踪目标：用户的一个提问“Spring Boot 是什么”**
    - **阶段 1：查询意图理解**
        - **核心任务**：代码有没有对用户的问题做预处理？(比如分词、去停用词、或者用 LLM 改写问题)。
    - **阶段 2：ES 查询构建 (最关键！)**
        - **核心任务**：在 Debug 模式下，拦截程序发给 ES 的 `Query DSL` (查询语句)。
        - *深度挖掘*：
            - 是否同时使用了 `match` (关键字检索) 和 `knn` (向量检索)？
            - 它们的权重 (`boost`) 分别是多少？
            - 有没有用 `min_score` 过滤掉相关度低的结果？
    - **阶段 3：Prompt 拼装**
        - **核心任务**：找到最后发给大模型的 Prompt 模板。
        - *深度挖掘*：如果检索回来的文档片段太多，超过了 Token 限制，代码里有没有做**截断**处理？
