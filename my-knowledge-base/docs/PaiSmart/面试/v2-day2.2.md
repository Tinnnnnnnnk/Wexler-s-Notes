# Pai-RAG Day 2 核心总结：异步解析与向量化入库

## 一、 系统异步解耦流程
* 文件分片合并完成后，系统会生成 `FileProcessingTask` 并异步投递至 Kafka 消息队列，实现上传与解析流程的解耦。
* 后台由 `FileProcessingConsumer` 监听专属 Topic 并执行消费任务。

## 二、 `downloadFileFromStorage` 的细节（文件下载的三道工程防线）
* **A. 防御 URL 过期（时间差隐患）：**
  防止因 Kafka 消息积压超过 1 小时，导致 MinIO 的预签名 URL (Presigned URL) 失效。若下载时遇到 `403 Forbidden` 状态码，系统会明确记录日志并抛出异常，进行精准拦截。
* **B. 防线程饥饿的超时控制：**
  严格设置了 HTTP 的连接超时（30秒）与读取超时（3分钟）。防止因存储节点的网络波动，导致某一个 Kafka 消费线程长时间阻塞挂起，最终耗尽微服务资源。
* **C. 透明存储路由设计：**
  具备极强的兼容性。先判断文件是否存在于本地磁盘（`file.exists()`），若存在则直接走本地 IO；若不存在，再判断是否走网络流下载。

## 三、 `parseAndSave` 的细节（防 OOM 与大模型语义保全）

### A. `checkMemoryThreshold`（全局内存熔断防线）
* 设置了内存安全检查机制，在解析前显式计算 JVM 的全局内存使用率。
* **防御性兜底：** 若内存占用超过设定的阈值（默认 `80%`），主动触发 `System.gc()`；若回收后仍超限，则抛出 `RuntimeException` 拒绝服务。这是一种“宁可牺牲单个任务，也要防止整个微服务 OOM 崩溃”的高可用架构思维。

### B. Tika & HanLP 流式多级语义切分
* **流式处理（阅后即焚）：** 禁用了 Tika 的全量加载，采用自定义的 `StreamingContentHandler`。系统一边解析一边在内存积攒文本，达到 `parentChunkSize`（1MB）就立刻触发切片入库，并清空内存缓冲区。
* **多级降级切分策略：** 为了兼顾 CPU 性能与大模型上下文语义，设计了“段落 -> 句子 -> 词语”的降级切分策略。绝大部分文本在段落和标点符号层面就被高效切分。
* **定向引入 AI 分词：** **只有遇到极其极端、无标点的超长难句时，才会定向调用 HanLP 智能中文分词器（`StandardTokenizer`）**。确保在不打爆 CPU 的前提下，精准保全上下文的核心名词不被生硬截断！

#### 具体切片降级细节：
1. **父块缓冲（主厨刀）：** Tika 一边解析，一边把文本塞进 `buffer`。只要 `buffer` 攒够了 1MB，才会触发一次子切片流程。
2. **段落切割（普通菜刀）：** 拿到这 1MB 文本后，系统首先用最廉价的字符串正则分割 `text.split("\n\n+")`，按**段落**把文本切开。
3. **句子切割（水果刀）：** 如果某个段落特别长（超过了限制的 `chunkSize`），系统会降级使用标点符号（。！？）把它切成**句子**。
4. **智能分词（终极狙击枪）：** 只有当遇到“一句话超过了 `chunkSize`，且中间一个标点符号都没有”的极端长句时，系统才会在 `splitLongSentence` 方法里，调用 HanLP 做智能词汇拆解。

## 四、 `bulkIndex` 的细节（极限 I/O 吞吐与可观测性）
* **高吞吐设计：** Elasticsearch 的 Bulk API 设计初衷是为了压榨网络 I/O 吞吐率，因此当我们一次性提交成百上千个文档时，Bulk 在底层是分发给各个分片独立处理的。
* **致命陷阱（静默数据丢失）：** 假设提交 1000 个文档，其中 3 个文档因 Mapping 格式错误而写入失败，**ES 网络层依然会返回 HTTP 200 成功状态码**。如果不加处理，就会造成大模型知识库的“盲区”。
* **代码防线一（打破 200 假象）：** 为了根治静默数据丢失问题，我强制校验了 Elasticsearch Java Client 提供的布尔值标志位（`response.errors()`），一旦为 true 即刻报错拦截。
* **代码防线二（精准追责与排障）：** 我在代码中设置了循环，去遍历 `response.items()` 数组。精准揪出具体是哪个 ID 的文档失败了，并记录其底层错误原因（`item.error().reason()`），这为后续的可观测性排障和死信队列重试提供了完美线索。

## 五、 附加面试核心武器：Kafka 架构深度拷问

### 1. 为什么最终选择了 Kafka 来实现异步处理？相比于 RabbitMQ 有什么优势？
* **流式处理与高吞吐：** 文件上传后涉及提取元数据、生成全文索引、AI 向量化等重度消耗资源的异步流式处理场景。Kafka 基于磁盘顺序写入，能处理每秒上百万条消息，非常适合这种高吞吐需求，这与 RabbitMQ 内存优先、读完即删的机制不同。
* **数据持久化与历史回溯（核心杀手锏）：** Kafka 的消息保存在磁盘上，可保留很久（如 7 天），消费者通过 offset 控制消费进度。这带来了极大的数据回溯优势：如果向量化模型升级或处理逻辑修复了 bug，只需新建消费组从历史 offset 开始消费，即可批量重跑老数据，无需触碰数据库。
* **生态扩展性：** Kafka 拥有极其成熟的生态（如 Kafka Connect、Kafka Streams、ksqlDB）。未来若需扩展实时计算和流式分析，可以直接在 Kafka 基础上平滑升级。

### 2. 在使用 Kafka 时，你是如何配置来确保消息的可靠传递的（防丢失）？
* **生产者端（防发送丢失）：** 设置了 `acks=all`，必须等待所有同步副本（ISR）写入成功才算发送成功，哪怕 Leader 宕机，只要 Follower 写入成功就不影响消息落盘，避免了“刚写完就丢”的风险。
* **服务端 Broker（防单点故障）：** Topic 副本数设为 3，并配置 `min.insync.replicas=2`。这相当于一道健康检查，如果存活副本不足 2 个则拒绝写入请求，防止写入到即将崩溃的分区上。
* **消费者端（防消费丢失）：** 放弃默认的自动提交，设置 `enable.auto.commit=false` 转为手动提交。只有当向量写入 ES 且数据库状态更新等业务逻辑真正成功后，才调用 `commitSync()` 提交 offset。这保证了“至少处理一次（At-least-once）”语义，系统崩溃重启也能从失败点拉取，绝不丢任务。

### 3. Kafka 怎么保证消息重试？（防毒药消息阻塞）
* **生产者网络重试：** 应对网络抖动或 Broker 临时故障，设置 `retries=3` 自动重试最多 3 次。配合 `enable.idempotence: true` 开启幂等性，确保重试时只会被写入一次，避免重复数据。
* **消费者业务重试与死信队列（DLQ）：** 面对业务失败（如 DB 超时、大模型 API 熔断），绝不能直接确认消息，也不能无限重试导致重复消费。系统中通过 `DefaultErrorHandler` 配置了优雅重试机制：利用 `FixedBackOff` 阻塞当前消费线程等待 1 秒后重试，最多重试 4 次。若 4 次后依然失败，则由 `DeadLetterPublishingRecoverer` 接管，将“毒药消息”路由至专用的死信队列（`file-processing-dlt`），触发人工排障并保证主队列畅通。
