## parseAndSave

### A.checkMemoryThreshold
- 设置了内存安全检查机制,假如JVM 内存超过了设定的阈值80%,那么就会触发垃圾回收机制
- 这使得我们有足够大的内存去处理文件,防止了OOM（OutOfMemoryError）异常
- 是防御性机制,在问题发生前就开始处理了,避免服务突然崩溃
### B.Tika&&HanLP 流式多级语义切分
- 禁用了 Tika 的全量读取，系统会一边解析一边在内存积攒，达到 1MB 就立刻触发切片并清空内存
- 为了兼顾 CPU 性能与大模型语义，我设计了‘段落 -> 句子 -> 词语’的多级降级切分策略。
- 绝大部分文本在段落和标点符号层面就被高效切分了；**只有遇到极其极端、无标点的超长难句时，我才会定向调用 HanLP 智能中文分词器**，确保在不打爆 CPU 的前提下，精准保全上下文的核心名词不被生硬截断！

#### 部分细节
- **父块缓冲：** Tika 一边解析，一边把文本塞进 `buffer`。只要 `buffer` 攒够了 `parentChunkSize`（默认 1MB），才会触发一次子切片流程。
- **段落切割：** 拿到这 1MB 文本后，系统首先用最廉价的字符串分割 `text.split("\n\n+")`，按**段落**把文本切开。
- **句子切割:**   如果某个段落特别长（超过了限制的 `chunkSize`），系统会继续用标点符号（。！？）把它切成 **句子** 。
- **只有当遇到那种“一句话超过了 `chunkSize`，且中间一个标点符号都没有”的极其变态的超长句时**，系统才会在 `splitLongSentence` 方法里，不得已地把这句话交给 HanLP (`StandardTokenizer.segment`) 去做智能词汇拆解

## bulkIndex 的细节
- Elasticsearch 的 Bulk API 设计初衷是为了高吞吐，因此如果我们给其很多个文档，Bulk会分开每一个单独处理
- 如果有1000个文档，其中3个文档出现了格式错误而导致写出失败，但是ES返回的依然是HTTP200
- 为了解决这个**静默数据丢失**的问题，我使用了Elasticsearch Java Client 专门提供的一个布尔值标志，来判断是否存在文档写入失败的情况
- 我还设置了一个循环去查询究竟是哪一些文档出现了格式错误，利于后续的排障
