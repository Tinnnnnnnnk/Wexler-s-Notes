# 信息安全基础紧凑讲义

## 文档用途
这份讲义不是给你“系统深造”用的，而是给你在较短时间内建立一个能汇报、能解释、不会太空的基础框架。

它的目标有三个：

- 帮你迅速搭起信息安全基础的主干
- 帮你区分最容易混淆的几个核心概念
- 帮你把“学过一点”变成“能说出逻辑和例子”

做到这四步后，就可以去做一版基础汇报了。

## 1. 整体框架
信息安全可以先看成一条完整链路：

`保护资产 -> 识别威胁 -> 找到漏洞 -> 理解攻击 -> 评估风险 -> 采取控制措施`

你当前最重要的不是背很多术语，而是先把这条链路吃透。

---

## 2. CIA 三要素
CIA 是信息安全最基础的三个目标。

### 2.1 机密性 Confidentiality
含义：

- 不该看的人看不到信息

例子：

- 学生成绩不能被无关同学查看
- 密码不能泄露
- 个人信息不能被随意导出

常见手段：

- 身份认证
- 权限控制
- 加密
- 脱敏
- 最小权限

一句话记忆：

`机密性 = 防泄露`

### 2.2 完整性 Integrity
含义：

- 数据不能被未授权地修改
- 如果被修改了，应该能发现

例子：

- 成绩不能被偷偷改高
- 传输中的文件不能被悄悄篡改
- 日志不能被随便删除或改写

常见手段：

- 哈希校验
- 数字签名
- 审计日志
- 权限控制
- 备份

一句话记忆：

`完整性 = 防乱改`

### 2.3 可用性 Availability
含义：

- 在需要使用的时候，系统和数据能正常使用

例子：

- 查成绩时系统不能一直打不开
- 服务器故障后数据还能恢复
- 系统不能被简单攻击打瘫

常见手段：

- 备份
- 冗余
- 容灾
- 监控告警
- 限流
- 抗拒绝服务能力

一句话记忆：

`可用性 = 不瘫痪`

### 2.4 为什么 CIA 重要
后面的很多安全措施，其实都是围绕这三个目标展开的。

例如：

- 加密主要偏向保护机密性
- 数字签名主要偏向保护完整性
- 备份主要偏向保护可用性

## 3. 资产、威胁、漏洞、攻击、风险、控制
这是最核心的一组概念，也是你汇报时最该讲清楚的部分。

### 3.1 资产 Asset
含义：

- 值得保护的东西

包括：

- 数据
- 账号
- 服务器
- 代码仓库
- 密钥
- 业务服务
- 声誉

一句话记忆：

`资产 = 你怕丢、怕泄露、怕被破坏的东西`

### 3.2 威胁 Threat
含义：

- 可能造成损害的事件或来源

例子：

- 黑客入侵
- 内部人员误操作
- 恶意软件
- 账号被盗
- 服务器断电
- 配置错误被利用

一句话记忆：

`威胁 = 可能发生的坏事`

### 3.3 漏洞 Vulnerability
含义：

- 系统中的弱点

例子：

- 弱密码
- 没打补丁
- 接口没有鉴权
- 文件上传校验不足
- 权限过大
- 配置不安全

一句话记忆：

`漏洞 = 系统的薄弱点`

### 3.4 攻击 Attack
含义：

- 威胁利用漏洞后，实际发起的动作

例子：

- 用弱密码暴力尝试登录
- 利用 SQL 注入读数据库
- 利用越权漏洞看别人的数据
- 上传恶意文件执行代码

一句话记忆：

`攻击 = 坏事真的动手发生了`

### 3.5 风险 Risk
含义：

- 某个威胁利用某个漏洞，对某个资产造成损失的可能性和影响

特别要注意：

- 只有漏洞，不等于高风险
- 只有威胁，不等于高风险
- 风险一定是结合资产、漏洞、威胁、后果一起看的

一句话记忆：

`风险 = 会不会出事 + 出事后有多严重`

### 3.6 控制 Control
含义：

- 用来降低风险的措施

例子：

- 多因素认证
- 补丁更新
- 权限控制
- 日志审计
- 监控
- 备份恢复

一句话记忆：

`控制 = 我们拿来降风险的办法`

### 3.7 一条完整例子
以“成绩管理系统”为例：

- 资产：成绩数据库
- 威胁：攻击者想窃取学生信息
- 漏洞：管理员密码太弱
- 攻击：撞库或暴力破解后台
- 风险：成绩和隐私数据泄露
- 控制：强密码、多因素认证、失败锁定、日志审计

这条完整链路你一定要会自己讲出来。

### 3.8 最容易混淆的区别
必须分清：

- 威胁：可能发生的坏事
- 漏洞：系统本身的弱点
- 攻击：坏事真正实施的过程
- 风险：造成损失的综合可能性和后果

最简单的比喻：

- 威胁是小偷
- 漏洞是门没锁
- 攻击是小偷进门
- 风险是东西被偷走

## 4. 认证、授权、最小权限
这一部分是“谁能进来、进来能干什么”。

### 4.1 认证 Authentication
含义：

- 证明“你是谁”

常见方式：

- 用户名密码
- 短信验证码
- 指纹
- 人脸
- 多因素认证

它解决的问题是：

`你是不是这个人`

### 4.2 授权 Authorization
含义：

- 决定“你能做什么”

例子：

- 学生只能看自己的成绩
- 老师只能改自己课程的成绩
- 管理员能管理全系统数据

它解决的问题是：

`你有权做哪些操作`

### 4.3 认证和授权的区别
这是非常经典的追问点。

标准答案：

- 认证确认身份
- 授权决定权限

一句很适合汇报时说的话：

`登录成功只说明认证通过，不代表对所有资源都有访问权限。`

### 4.4 最小权限 Least Privilege
含义：

- 只给完成任务所需的最低权限，不多给

为什么重要：

- 降低误操作影响
- 降低账号被盗后的损失
- 降低权限滥用风险

例子：

- 普通学生不应有管理员权限
- 开发人员不应默认能看全部生产数据
- 脚本不应默认以 root 身份运行

一句话记忆：

`最小权限 = 够用就行，不多给`

### 4.5 RBAC 和 ABAC
你现阶段只需要知道大概区别。

RBAC：

- 基于角色分配权限
- 例如学生、老师、管理员

ABAC：

- 基于属性分配权限
- 例如“本课程教师”才能改“本课程成绩”

简化理解：

- RBAC 更简单
- ABAC 更灵活

## 5. 哈希、加密、数字签名
这三个最容易混，必须分清它们分别解决什么问题。

### 5.1 哈希 Hash
主要作用：

- 检查内容有没有变
- 安全存储密码摘要

特点：

- 输入任意长度
- 输出固定长度
- 不可逆
- 输入稍微变一点，结果会差很多

用途：

- 存密码哈希
- 校验文件完整性
- 比较消息是否被篡改

关键点：

`哈希不是为了把内容藏起来，而是为了做校验或摘要。`

### 5.2 加密 Encryption
主要作用：

- 保护内容不被别人看懂

特点：

- 可逆
- 依赖密钥
- 核心目的是保密

分两类：

对称加密：

- 加密解密用同一把密钥
- 速度快
- 适合大数据量
- 例子：AES

非对称加密：

- 公钥和私钥成对出现
- 速度较慢
- 适合密钥交换和身份相关场景
- 例子：RSA、ECC

关键点：

`加密的核心是保密，内容之后可以解回来。`

### 5.3 数字签名 Digital Signature
主要作用：

- 证明消息是谁发的
- 证明消息没有被改

基本思路：

- 发送方用私钥签名
- 接收方用公钥验证

它主要提供：

- 来源可信
- 内容完整

关键点：

`数字签名不是主要为了保密，而是为了证明来源和防篡改。`

### 5.4 三者的快速区分
这是你必须能脱口而出的版本：

- 哈希：看内容有没有变化
- 加密：把内容藏起来
- 数字签名：证明是谁发的、内容有没有被改

### 5.5 一个完整例子
老师上传成绩文件时：

- 为了防泄露：使用加密
- 为了防篡改：做哈希校验
- 为了证明是老师本人提交：使用数字签名

这个例子非常适合汇报时使用。

## 6. 你真正需要建立的，不是知识点，而是关系
你现在阶段最重要的是把下面关系建立起来：

- CIA 是安全目标
- 资产是要保护的对象
- 威胁和漏洞结合后形成风险
- 攻击是具体利用过程
- 控制是降低风险的方法
- 认证和授权控制“谁进来、能做什么”
- 哈希、加密、数字签名分别保护数据的不同方面

只要这些关系你讲顺了，你的汇报就不会显得太空。

## 7. 你至少应该会说的 6 句话
如果这六句你能自然讲出来，就已经达到“基础汇报可用”水平。

1. 信息安全可以围绕机密性、完整性、可用性三个目标来理解。
2. 威胁、漏洞、攻击和风险是相关但不同的概念。
3. 风险是威胁利用漏洞对资产造成损失的可能性和影响。
4. 认证解决“你是谁”，授权解决“你能做什么”。
5. 最小权限原则可以降低误操作和账号失陷后的影响。
6. 哈希、加密和数字签名分别解决校验、保密和身份证明的问题。
