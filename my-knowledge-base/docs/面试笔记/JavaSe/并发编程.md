### 线程和进程的区别是什么
- 进程是操作系统分配资源的最小单位
- 线程是进程中的独立执行单元，多个线程共享同一个进程的资源
### 协程是什么
- 协程是比线程更轻量级的并发单元，可以在单线程中实现并发执行
- 协程在用户态进行调度，避免了线程切换的内核态转换开销
- JAVA本身不支持协程，可以使用Quasar、Kotlin等框架来实现
### 线程之间如何通信
- 使用volatile 关键字 和 synchronized 关键字 访问和修改同一对象 ， 从而实现信息的传递
	- volatile：告知程序 任何对于该变量的访问都要从共享内存中读取，并同步刷新回共享内存。修饰成员变量
	- synchronized：确保多个线程同一时刻只有一个线程在执行方法或代码块。修饰方法或代码块
- wait( ) 和 notify( )方法
	- 使用 共享对象的 wait( )方法，会使该线程进入该对象的等待池，释放持有的锁，进入等待态
	- 使用 共享对象的 notify( )方法，该对象会唤醒等待池中的一个线程，进入锁池，等待获取锁
### 线程有哪些创建方式
- 重写父类Thread 的 run( )方法
  ```java
  class ThreadTask extends Thread {
      public void run() {
          System.out.println("看完二哥的 Java 进阶之路，上岸了!");
      }

      public static void main(String[] args) {
          ThreadTask task = new ThreadTask();
          task.start();
      }
  }
  ```
	- 调用 start( )方法来启动线程
	- 如果ThreadTask 继承了另外一个类，那么就不能再继承 Thread 类，因为JAVA 不支持多重继承
- 重写Runnable接口的 run( ) 方法
  ```java
  class RunnableTask implements Runnable {
      public void run() {
          System.out.println("看完二哥的 Java 进阶之路，上岸了!");
      }

      public static void main(String[] args) {
          RunnableTask task = new RunnableTask();
          Thread thread = new Thread(task);
          thread.start();
      }
  }
  ```
	- 将实现类的对象作为参数传递给Thread对象的构造方法，最后调用start( )方法启动线程
	- 避免了JAVA的单继承限制
- 重写Callable接口的 call( )方法
  ```java
  class CallableTask implements Callable<String> {
      public String call() {
          return "看完二哥的 Java 进阶之路，上岸了!";
      }

      public static void main(String[] args) throws ExecutionException, InterruptedException {
          CallableTask task = new CallableTask();
          FutureTask<String> futureTask = new FutureTask<>(task);
          Thread thread = new Thread(futureTask);
          thread.start();
          System.out.println(futureTask.get());
      }
  }
  ```
	- 创建FutureTask对象，参数为Callable实现类的对象，紧接着创建Thread对象，参数为FutureTask对象，最后调用start( )方法启动线程
	- 可以获取线程执行的结果

### 调用 start 方法会执行 run方法，为什么不直接调用run 方法
- 调用 start 方法会创建一个新的线程，并异步执行 run方法中的代码
- 如果直接调用了 run 方法，那么只会在当前的主线程中执行run 方法中的所有代码，没有创建新的线程，也就达不到多线程并发的目的

### sleep 和 wait 有什么区别
- 所属类不同，sleep属于 Thread类，wait属于 Object类
- 行为不同，一个线程在持有某个对象锁的时候调用 sleep方法，在睡眠的时候还是会持有这个锁；而如果此时调用了wait方法，这个线程会释放这个对象锁
- 使用条件不同，sleep可以在任何地方被调用；wait必须在同步方法或同步代码块中被调用，调用的前提是当前线程必须持有对象锁，否则会抛出 IllegalMonitorStateException 异常
- 唤醒方式不同，调用sleep后等在指导的时间内会暂停执行，之后就可以等待cpu 调度执行了；调用wait后需要在等待池等其他线程在同一对象上调用notify方法后，才能再次获得CPU执行权

### ThreadLocal 是什么
- ThreadLocal是 线程本地变量 ，实现线程局部变量的工具类，从而实现线程隔离
- 使用步骤
  - ①创建ThreadLocal
    ```java
    // 创建一个ThreadLocal变量
    public static ThreadLocal<String> localVariable = new ThreadLocal<>();
    ```
  - ②设置ThreadLocal的值
    ```java
    // 设置ThreadLocal变量的值
    localVariable.set("test");
    ```
  - ③获取ThreadLocal的值
    ```java
    // 获取ThreadLocal变量的值
    String value = localVariable.get();
    ```
  - ④删除ThreadLocal的值
    ```java
    // 删除ThreadLocal变量的值
    localVariable.remove();
    ```
### ThreadLocal的优点
- 每个线程访问的变量副本都是独立的，避免共享变量引起的线程安全问题
- ThreadLocal可用于跨方法、跨类传递上下文数据，不需要在方法间传递参数
### ThreadLocal 的实现
- 我们创建一个ThreadLocal对象并调用set方法时，就是在当前线程中初始化了一个ThreadLocalMap
- ThreadLocalMap是ThreadLocal的一个静态内部类，内部维护了一个 Entry 数组， Key是ThreadLocal对象，value是线程的局部变量，也就是每个线程维护一个 Map
- set方法就是将对象存入 Map，get 方法就是从 Map中取出对象，Map的大小由ThreadLocal的对象数量决定
### 强引用和弱引用 与 ThreadLocal 内存泄露
- 强引用
	- 比如平时的 Object obj=new Object()
	- 除非被置为 null 或者超出作用域，否则永远不会被JVM 垃圾回收器回收
	- 即使内存不足，JVM 抛出OutOfMemoryError，也不会回收
- 弱引用
	- 通过 WeakReference 类来实现
	- JVM进行垃圾回收时，只要发现弱引用对象，就直接回收
- ThreadLocal 内存泄露
	- 在ThreadLocalMap中，key是弱引用，value是强引用
	- 如果一个线程一直运行， 且value 一直指向一个强引用对象，那么这个对象不会被回收，从而导致内存泄露
	- 我们可以在使用完ThreadLocal后及时调用remove去释放内存,也就是会遍历TheardLocalMap，将当前key为当前ThreadLocal的Entry去调用clear，再将value设置为null，之后再清除key为null的Entry

### ThreadLocalMap 怎么解决Hash 冲突
- 使用开放定址法
	- 计算的 槽位i 如果被占用，就去寻找i+1；如果i+1被占用，就去寻找i+2，直到找到一个空位为止
- 为什么不用别的？
	- ThreadLocalMap设计的目的是存储线程私有数据，而不是存放大量Key，线性探测法更省空间；拉链法还需要额外维护一个链表甚至红黑树

### Java内存模型
- Java 内存模型 是 Java 虚拟机规范中定义的一个抽象模型，用来描述多线程环境中共享变量的内存可见性
- 共享变量存储在主内存中，每个线程都有一个 私有的本地内存，存储了共享变量的副本
	- 当一个线程更改了本地内存中共享变量的副本，需要JVM同步刷新到主内存中，确保其他线程可以看到更改
	- 当一个线程需要读取共享变量，首先会在本地内存中读取，如果本地内存的副本过时了，JVM就会将主内存中的共享变量最新值刷新到本地内存中

### volatile 关键字
- 用来保证可见性：线程修改volatile变量后，其他线程可以立马看到最新值
- 防止指令重排序，volatile变量 的写入 不会被重排序到它之前的代码，也就是重排序不能修改 与 volatile 变量有关的代码
#### 如何保证可见性？
- 当线程对 volatile 变量进行写操作的时候，JVM会给这个变量写入后插入一个 写屏障指令，强制将这个本地内存的变量值刷新到主内存中
- 当线程对 volatile 变量进行读操作时，JVM 会插入一个读屏障指令，这个指令会强制让本地内存中的变量值失效，从而重新从主内存中读取最新的值。

#### 如何保证有序性？
- 会在读写操作前后 添加内存屏障，来禁止一些读写操作 和重排

### Synchronized 锁升级
- 为了提升synchronized的性能
- 没有线程竞争的时候 采用 低开销的 偏向锁，此时没有额外的 CAS 操作
- 轻度竞争时 使用 轻量级锁 ，采用 CAS自旋，避免线程阻塞
- 重度竞争时 使用 重量级锁 ，由Monitor 机制实现，需要线程阻塞

### Synchronized 四种锁
- 无锁状态：
	- 对象未被锁定，Mark Word 存储对象的哈希码等信息
- 偏向锁：
	- 当线程第一次获取锁时，进入偏向模式，JVM 会在 Mark Word 中记录该线程的 ID。后续如果**还是这个线程**来获取锁，无需进行任何额外的加锁操作，直接进入同步代码块，性能极高。
- 轻量级锁：
	- 当有其他线程来尝试获取锁，偏向锁就会失效，升级为轻量级锁。它适用于**竞争比较轻微、线程交替执行**的场景。此时，未拿到锁的线程不会立刻去睡觉（阻塞），而是通过 **CAS 自旋** 在原地稍作等待，期待锁马上被释放。
- 重量级锁：
	- 如果竞争变得非常激烈（比如自旋超过了一定次数，或者自旋的时候又冒出来第三个线程来抢锁），轻量级锁就会膨胀为重量级锁。这时候再自旋就是白白浪费 CPU 了，所以其他等待的线程会**进入阻塞状态，交由操作系统来调度**。、
- 最开始 **锁对象** 是无锁状态。当线程第一次获取锁的时候，这个对象就变成了**偏向锁**；此时如果有别的线程尝试来获取锁，那么锁就会升级为**轻量级锁**，没拿到锁的线程就会进行 **CAS 自旋**；如果线程自旋超过一定次数，或者又来了第三个线程来抢锁，那么轻量级锁就会膨胀成**重量级锁**，其他等待的线程就会进入**阻塞状态**。
### Volatile 与 Synchronized 的区别
- volatile 用于修饰变量， 确保该变量的更新操作 对于所有线程都是可见的，一旦某一个线程修改了volatile变量，其他所有的线程都可以看到最新值
- synchronized用于修饰方法和代码块，实现了互斥访问，确保同一时刻只有一个线程可以执行该方法或代码块

### Synchronized 和 ReentrantLock
- synchronized是Java的内置关键字，而ReentrantLock是基于AQS实现的
- synchronized 可以实现自动加锁自动解锁，而ReentrantLock需要手动使用lock 和 unlock
- synchronized 只可以通过wait和 notify 唤醒，属于单路通知； ReentrantLock可以实现多路通知，绑定多个Condition
- synchronized 可以在方法和代码块上加锁，而 ReentrantLock只可以在代码块上加锁，但可以指定是公平锁还是非公平锁
- ReentrantLock 提供了一种能够中断等待锁的线程机制，通过 `lock.lockInterruptibly()` 来实现。

### 并发量大的场景下，使用 synchronized 还是reentrantLock
- 选择 reentrantlock
- 提供了超时获取锁的特性（`tryLock`）。这种机制可以有效防止大量线程在抢锁时无限期阻塞，从而避免高并发下内存耗尽引发的**服务雪崩**。
- 支持条件变量（`Condition`）。 相比于 `synchronized` 粗暴的 `notifyAll`，`Condition` 可以实现更友好的、**精确的线程间通信**，能有效避免‘惊群效应’带来的 CPU 资源浪费。
- 允许更灵活、更细粒度的锁控制。比如结合业务实现分段锁机制，能极大减少并发情况下的锁竞争冲突。

### Lock 和 ReentrantLock的lock方法实现逻辑
- Lock是JUC中的一个接口
- lock方法具体实现由ReentrantLock的内部类sync来实现
- lock方法会先通过CAS获取锁，如果当前锁没有被持有就可以获取，否则就将当先线程加入到AQS等待队列中
### AQS
**1. 基本定位**
- 它是一个**抽象类**，全称叫 **AbstractQueuedSynchronizer（抽象队列同步器）**，是 Java 并发包（JUC）中各种锁和同步组件的底层基石。
**2. 核心数据结构（状态 + 队列）**
- **`state` 变量：** 内部使用了一个 `volatile int` 类型的 `state` 变量来记录当前锁的占用状态。
- **双端队列：** 内部维护了一个 FIFO（先进先出）的双向链表队列，里面存储的是将等待线程包装起来的 **Node 节点**。
**3. 核心工作机制（排队与唤醒）**
- 当线程尝试获取锁（修改 `state`）失败时，AQS 就会将这个线程包装成 Node 节点，加入到双端队列的尾部，并使线程进入阻塞挂起状态。
- 当持有锁的线程释放资源时，AQS 会负责从双端队列的**队首（头部）**唤醒下一个处于等待状态的节点，让它去重新竞争锁。
**4. 两种工作模式（扩展性极强）**
- **独占模式：** 同一时刻只允许一个线程拥有锁（如 `ReentrantLock`）。
- **共享模式：** 允许多个线程同时持有锁，具体允许的数量可以由实现类自定义（如 `Semaphore` 信号量、`CountDownLatch` 倒计时器）。
### 公平锁和非公平锁
- 公平锁是多线程竞争锁时，先来先服务，非公平锁是当锁被释放时，任何请求锁的线程都有机会获取锁
- 在创建 ReentrantLock 时，传递参数 true 生成公平锁，不传递或者传递false生成非公平锁
### CAS
- CAS 是一种乐观锁，用于比较一个变量的主内存值是否等于预期值，如果相等，则更新值，否则重试。
- 有三个值
	- V：要更新的值（var）
	- E：预期值（expected）
	- N：新值（new）
- 上述三个值的比较和更新需要原子的不可中断的，CAS的原子性由Unsafe保证
- 比如AtomicInteger类的compareAndSet就是一个CAS方法，调用的是Unsafe的compareAndSet
- CPU会发出一个LOCK指令来进行总线锁定，防止其他CPU对内存地址进行操作，直到当前指令执行完成。
### CAS 自身存在的问题
**1.自旋开销大
- CAS失败时会不断自旋重试，可以设置一个自旋次数限制，超过一定次数就可以让Synchronized挂起线程
**2.涉及到多个变量同时更新
- 将多个变量封装成一个对象，使用AtomicReference 进行CAS更新
**3.ABA问题
- ABA 问题指的是，一个值原来是 A，后来被改为 B，再后来又被改回 A，这时 CAS 会误认为这个值没有发生变化。
- 可以使用版本号/时间戳的方式来解决 ABA 问题。sssssss

### 自旋锁
- 当线程获取锁失败时，不立即阻塞，而是通过自旋，循环等待的方式不断尝试获取锁
- 可以避免切换线程带来的开销，但代价是如果锁被占用时间过长，会导致线程空转，浪费CPU资源
### 乐观锁和悲观锁
- 乐观锁认为冲突不会发生，在操作前不加锁，只有在更新数据时才会检查数据是否被修改，如果数据被修改了，就会重试，直到重试成功或者重试次数达到上限
- 悲观锁认为每次访问共享资源都会发生冲突，所以在操作前加锁，防止其他线程修改数据


### 常用的并发容器
#### CopyOnWriteArrayList 的实现原理
- 是ArrayList的线程安全版本，适用于 读多写少且能容忍短时间内读到脏数据 的场景
- CopyOnWrite 的核心思想是写操作时创建一个新数组， 修改后再替换原数组，确保读操作无锁
- 内部使用volatile 变量来修饰array，确保读操作的内存可见性
- 写操作通过ReentrantLock保证线程安全
- 写操作可能会因为数组过大而性能受影响

### 线程池
#### 什么是线程池
- 用来管理和复用线程的工具
- ThreadPoolExecutor是线程池的核心体现
- 用一家餐厅来类比，线程池里的 核心线程 就像是餐厅里的正式员工，当有客人（任务）来的时候，正式员工就上去服务；当客人太多，员工忙不过来，餐厅就会让客人在等待区（阻塞队列）排队；如果等候区也满了，餐厅会临时招聘一些 临时工（非核心线程）来帮忙，如果临时工也招满， 那么餐厅只能拒绝新客人（拒绝策略）
#### 线程池的工作流程？
- 任务提交 ——核心线程执行——任务队列缓存——非核心线程执行——拒绝策略处理
- 1：线程池提交任务
- 2：线程池会创建核心线程来执行任务
- 3：如果核心线程都在忙，任务就会放到任务队列（阻塞队列）中
- 4：如果任务队列满了，当前线程数小于最大线程数，就会创建新的线程来处理任务
- 5：如果当前线程数超过了最大线程数，且任务队列已满，那么线程池会执行拒绝策略
### 线程池的主要参数
- corePoolSize
	- 核心线程数
- maximumPoolSize
	- 线程池允许的最大线程数
- workQueue
	- 任务队列，阻塞队列
- handler
	- 拒绝策略，也就是线程数达到maximumPoolSize，任务队列也满了的时候的处理方式
- threadFactory
	- 线程工厂，创建线程
- keepAliveTime
	- 非核心线程的存活时间，时间到了就销毁
- unit
	- keepAliveTime的时间单位
#### 线程池的拒绝策略
- AbortPolicy
	- 默认的拒绝策略，会抛出 RejectedExecutionException 异常
- CallerRunsPolicy
	- 让提交任务的线程自己执行这个任务
- DiscardOldestPolicy
	- 等待队列会丢弃队列中等待最久的一个任务，然后尝试重新提交被拒绝的任务
- DiscardPolicy
	- 直接丢弃被拒绝的任务

##### 1. 固定大小线程池 (FixedThreadPool)
> [!note] ⚙️ `Executors.newFixedThreadPool(int nThreads)`
- **运行机制：** 核心线程数 = 最大线程数。池子里的线程数量永远是固定的，没有非核心（临时）线程。
- **适用场景：** 用于任务数量确定，并且对线程数有明确要求的场景。
- **底层队列：** `LinkedBlockingQueue`（无界队列）。
- **⚠️ 面试避坑点：** 因为使用的是无界队列，如果任务堆积速度远大于处理速度，会导致队列无限变长，最终撑爆内存引发 **OOM (Out Of Memory)**。

##### 2. 缓存线程池 (CachedThreadPool)
> [!tip] 🚀 `Executors.newCachedThreadPool()`
- **运行机制：** 核心线程数 = 0，最大线程数 = `Integer.MAX_VALUE`。全都是临时工，谁闲着谁上，不够就一直招新线程。空闲超过 60 秒就被回收。
- **适用场景：** 用于短时间内任务量波动大，且每个任务执行时间很短的场景。
- **底层队列：** `SynchronousQueue`（同步移交队列，容量为 0，不存任务，直接把任务塞给空闲线程）。
- **⚠️ 面试避坑点：** 极端并发下，如果任务执行极其缓慢，它会疯狂创建新线程（上限高达 21 亿），最终导致 CPU 瘫痪或 **OOM**。

##### 3. 单线程线程池 (SingleThreadExecutor)
> [!info] 🚦 `Executors.newSingleThreadExecutor()`
- **运行机制：** 核心线程数 = 1，最大线程数 = 1。整个池子里永远只有一位孤勇者在干活。
- **适用场景：** 用于需要按顺序串行执行任务的场景（绝对保证任务的先来后到）。
- **底层队列：** `LinkedBlockingQueue`（无界队列）。
- **⚠️ 面试避坑点：** 同样因为无界队列，排队的任务太多时也会导致 **OOM**。

##### 4. 定时任务线程池 (ScheduledThreadPool)
> [!abstract] ⏱️ `Executors.newScheduledThreadPool(int corePoolSize)`
- **运行机制：** 核心线程数固定，最大线程数为 `Integer.MAX_VALUE`。支持定时执行或周期性执行任务。
- **适用场景：** 用于需要定时执行、延迟执行或者周期性轮询任务的场景。
- **底层队列：** `DelayedWorkQueue`（延迟阻塞队列）。

---

> [!danger] 🌟 面试终极必杀问：为什么《阿里巴巴 Java 开发手册》强制禁止使用 `Executors` 快捷创建线程池？
> **满分实战回答：**
> “因为这四种内置工厂方法都存在极大的 **OOM（内存溢出）隐患**。
> `Fixed` 和 `Single` 采用了无界队列，任务极易无限堆积；
> `Cached` 和 `Scheduled` 允许创建的最大线程数是 Integer 的最大值，极易耗尽系统资源。
> 在真实生产环境中，为了安全可控，我们必须手动通过 `ThreadPoolExecutor` 去创建，明确指定队列容量并配备合理的拒绝策略！”


#### 线程池调优
- 首先我会根据任务类型设置核心线程数参数，比如IO密集型任务会设置为CPU核心数* 2的经验值
- 其次我会结合线程池动态调整的能力，在流量波动时通过 setCorePoolSize 平滑扩容，或者直接使用 DynamicTp 实现线程池参数的自动化调整。
- 最后，我会通过内置的监控指标建立容量预警机制。比如通过 JMX 监控线程池的运行状态，设置阈值，当线程池的任务队列长度超过阈值时，触发告警。
#### 线程池使用的时候需要注意什么
- 1：选择一个合适的线程池，线程池过小可能会导致任务一直排队；线程池过大可能会导致大家都在竞争CPU资源，增加切换上下文的开销
- 2：选择一个合适的任务队列。使用有界队列可以避免资源被耗尽，但有可能导致任务被拒绝；使用无界队列可以避免任务被拒绝，但是会有内存耗尽OOM的风险
- 3：尽量使用自定义的线程池；因为 newFixedThreadPool 线程池由于使用了 LinkedBlockingQueue，队列的容量默认无限大，任务过多时会导致内存溢出；newCachedThreadPool 线程池由于核心线程数无限大，当任务过多的时候会导致创建大量的线程，导致服务器负载过高宕机。

#### 线程池执行中断电怎么办
- 线程池本身只能在内存中调度，不会持久化
- 我会考虑
	- 持久化任务，将任务持久化到数据库或者消息队列中，等通电了再重新执行
	- 任务幂等性，也就是任务无论执行多少次，结果一致
	- 恢复策略，等系统重启后，有一个恢复流程去检查上次是否有未完成任务，并重新加载到线程池中执行
### Java的常见并发关键字
- **synchronized**：用于方法或者代码块，确保同一时间只有一个线程可以执行被 synchronized 修饰的代码，适用于保护共享资源的访问。
- **volatile**：用于变量，确保变量的可见性，防止指令重排序，适用于状态标志等场景。
### 并发容器有哪些
- **ConcurrentHashMap**：线程安全的哈希表，支持高并发读写
- **CopyOnWriteArrayList**：写时复制的ArrayList，适用于读多血少的场景，会有短时间不同步的问题
- **BlockingQueue**：阻塞队列，用于生产者消费者场景
- **ConcurrentLinkedQueue**：非阻塞的线程安全队列。
- **ConcurrentSkipListMap**：基于跳表实现的线程安全有序 Map。
