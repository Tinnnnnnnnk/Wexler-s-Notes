### 线程和进程的区别是什么
- 进程是操作系统分配资源的最小单位
- 线程是进程中的独立执行单元，多个线程共享同一个进程的资源
### 协程是什么
- 协程是比线程更轻量级的并发单元，可以在单线程中实现并发执行
- 协程在用户态进行调度，避免了线程切换的内核态转换开销
- JAVA本身不支持协程，可以使用Quasar、Kotlin等框架来实现
### 线程之间如何通信
- 使用volatile 关键字 和 synchronized 关键字 访问和修改同一对象 ， 从而实现信息的传递
	- volatile：告知程序 任何对于该变量的访问都要从共享内存中读取，并同步刷新回共享内存。修饰成员变量
	- synchronized：确保多个线程同一时刻只有一个线程在执行方法或代码块。修饰方法或代码块
- wait( ) 和 notify( )方法
	- 使用 共享对象的 wait( )方法，会使该线程进入该对象的等待池，释放持有的锁，进入等待态
	- 使用 共享对象的 notify( )方法，该对象会唤醒等待池中的一个线程，进入锁池，等待获取锁
### 线程有哪些创建方式
- 重写父类Thread 的 run( )方法
  ```java
  class ThreadTask extends Thread {
      public void run() {
          System.out.println("看完二哥的 Java 进阶之路，上岸了!");
      }

      public static void main(String[] args) {
          ThreadTask task = new ThreadTask();
          task.start();
      }
  }
  ```
	- 调用 start( )方法来启动线程
	- 如果ThreadTask 继承了另外一个类，那么就不能再继承 Thread 类，因为JAVA 不支持多重继承
- 重写Runnable接口的 run( ) 方法
  ```java
  class RunnableTask implements Runnable {
      public void run() {
          System.out.println("看完二哥的 Java 进阶之路，上岸了!");
      }

      public static void main(String[] args) {
          RunnableTask task = new RunnableTask();
          Thread thread = new Thread(task);
          thread.start();
      }
  }
  ```
	- 将实现类的对象作为参数传递给Thread对象的构造方法，最后调用start( )方法启动线程
	- 避免了JAVA的单继承限制
- 重写Callable接口的 call( )方法
  ```java
  class CallableTask implements Callable<String> {
      public String call() {
          return "看完二哥的 Java 进阶之路，上岸了!";
      }

      public static void main(String[] args) throws ExecutionException, InterruptedException {
          CallableTask task = new CallableTask();
          FutureTask<String> futureTask = new FutureTask<>(task);
          Thread thread = new Thread(futureTask);
          thread.start();
          System.out.println(futureTask.get());
      }
  }
  ```
	- 创建FutureTask对象，参数为Callable实现类的对象，紧接着创建Thread对象，参数为FutureTask对象，最后调用start( )方法启动线程
	- 可以获取线程执行的结果

### 调用 start 方法会执行 run方法，为什么不直接调用run 方法
- 调用 start 方法会创建一个新的线程，并异步执行 run方法中的代码
- 如果直接调用了 run 方法，那么只会在当前的主线程中执行run 方法中的所有代码，没有创建新的线程，也就达不到多线程并发的目的
### 线程的常用调度方法
- stop方法
	- stop方法用来强制停止线程，可能会在不一致的状态下释放锁，因此被废弃了
- interrupt方法
	- interrupt方法用来通知线程停止，但不会直接停止，会让线程自己去停止

### sleep 和 wait 有什么区别
- 所属类不同，sleep属于 Thread类，wait属于 Object类
- 行为不同，一个线程在持有某个对象锁的时候调用 sleep方法，在睡眠的时候还是会持有这个锁；而如果此时调用了wait方法，这个线程会释放这个对象锁
- 使用条件不同，sleep可以在任何地方被调用；wait必须在同步方法或同步代码块中被调用，调用的前提是当前线程必须持有对象锁，否则会抛出 IllegalMonitorStateException 异常
- 唤醒方式不同，调用sleep后等在指导的时间内会暂停执行，之后就可以等待cpu 调度执行了；调用wait后需要在等待池等其他线程在同一对象上调用notify方法后，才能再次获得CPU执行权

### ThreadLocal 是什么
- ThreadLocal是 线程本地变量 ，实现线程局部变量的工具类，从而实现线程隔离
- 使用步骤
  - ①创建ThreadLocal
    ```java
    // 创建一个ThreadLocal变量
    public static ThreadLocal<String> localVariable = new ThreadLocal<>();
    ```
  - ②设置ThreadLocal的值
    ```java
    // 设置ThreadLocal变量的值
    localVariable.set("test");
    ```
  - ③获取ThreadLocal的值
    ```java
    // 获取ThreadLocal变量的值
    String value = localVariable.get();
    ```
  - ④删除ThreadLocal的值
    ```java
    // 删除ThreadLocal变量的值
    localVariable.remove();
    ```
### ThreadLocal的优点
- 每个线程访问的变量副本都是独立的，避免共享变量引起的线程安全问题
- ThreadLocal可用于跨方法、跨类传递上下文数据，不需要在方法间传递参数
### ThreadLocal 的实现
- 我们创建一个ThreadLocal对象并调用set方法时，就是在当前线程中初始化了一个ThreadLocalMap
- ThreadLocalMap是ThreadLocal的一个静态内部类，内部维护了一个 Entry 数组， Key是ThreadLocal对象，value是线程的局部变量，也就是每个线程维护一个 Map
- set方法就是将对象存入 Map，get 方法就是从 Map中取出对象，Map的大小由ThreadLocal的对象数量决定
### 强引用和弱引用 与 ThreadLocal 内存泄露
- 强引用
	- 比如平时的 Object obj=new Object()
	- 除非被置为 null 或者超出作用域，否则永远不会被JVM 垃圾回收器回收
	- 即使内存不足，JVM 抛出OutOfMemoryError，也不会回收
- 弱引用
	- 通过 WeakReference 类来实现
	- JVM进行垃圾回收时，只要发现弱引用对象，就直接回收
- ThreadLocal 内存泄露
	- 在ThreadLocalMap中，key是弱引用，value是强引用
	- 如果一个线程一直运行， 且value 一直指向一个强引用对象，那么这个对象不会被回收，从而导致内存泄露
	- 我们可以在使用完ThreadLocal后及时调用remove去释放内存,也就是会遍历TheardLocalMap，将当前key为当前ThreadLocal的Entry去调用clear，再将value设置为null，之后再清除key为null的Entry

### ThreadLocalMap 怎么解决Hash 冲突
- 使用开放定址法
	- 计算的 槽位i 如果被占用，就去寻找i+1；如果i+1被占用，就去寻找i+2，直到找到一个空位为止
- 为什么不用别的？
	- ThreadLocalMap设计的目的是存储线程私有数据，而不是存放大量Key，线性探测法更省空间；拉链法还需要额外维护一个链表甚至红黑树

### Java内存模型
- Java 内存模型 是 Java 虚拟机规范中定义的一个抽象模型，用来描述多线程环境中共享变量的内存可见性
- 共享变量存储在主内存中，每个线程都有一个 私有的本地内存，存储了共享变量的副本
	- 当一个线程更改了本地内存中共享变量的副本，需要JVM同步刷新到主内存中，确保其他线程可以看到更改
	- 当一个线程需要读取共享变量，首先会在本地内存中读取，如果本地内存的副本过时了，JVM就会将主内存中的共享变量最新值刷新到本地内存中

### volatile 关键字
- 用来保证可见性：线程修改volatile变量后，其他线程可以立马看到最新值
- 防止指令重排序，volatile变量 的写入 不会被重排序到它之前的代码，也就是重排序不能修改 与 volatile 变量有关的代码
#### 如何保证可见性？
- 当线程对 volatile 变量进行写操作的时候，JVM会给这个变量写入后插入一个 写屏障指令，强制将这个本地内存的变量值刷新到主内存中
- 当线程对 volatile 变量进行读操作时，JVM 会插入一个读屏障指令，这个指令会强制让本地内存中的变量值失效，从而重新从主内存中读取最新的值。

#### 如何保证有序性？
- 会在读写操作前后 添加内存屏障，来禁止一些读写操作 和重排

### Synchronized 锁升级
- 为了提升synchronized的性能
- 没有线程竞争的时候 采用 低开销的 偏向锁，此时没有额外的 CAS 操作
- 轻度竞争时 使用 轻量级锁 ，采用 CAS自旋，避免线程阻塞
- 重度竞争时 使用 重量级锁 ，由Monitor 机制实现，需要线程阻塞

### Synchronized 四种锁
- 无锁状态：
	- 对象未被锁定，Mark Word 存储对象的哈希码等信息
- 偏向锁：
	- 当线程第一次获取锁时，进入偏向模式，JVM 会在 Mark Word 中记录该线程的 ID。后续如果**还是这个线程**来获取锁，无需进行任何额外的加锁操作，直接进入同步代码块，性能极高。
- 轻量级锁：
	- 当有其他线程来尝试获取锁，偏向锁就会失效，升级为轻量级锁。它适用于**竞争比较轻微、线程交替执行**的场景。此时，未拿到锁的线程不会立刻去睡觉（阻塞），而是通过 **CAS 自旋** 在原地稍作等待，期待锁马上被释放。
- 重量级锁：
	- 如果竞争变得非常激烈（比如自旋超过了一定次数，或者自旋的时候又冒出来第三个线程来抢锁），轻量级锁就会膨胀为重量级锁。这时候再自旋就是白白浪费 CPU 了，所以其他等待的线程会**进入阻塞状态，交由操作系统来调度**。、
- 最开始 **锁对象** 是无锁状态。当线程第一次获取锁的时候，这个对象就变成了**偏向锁**；此时如果有别的线程尝试来获取锁，那么锁就会升级为**轻量级锁**，没拿到锁的线程就会进行 **CAS 自旋**；如果线程自旋超过一定次数，或者又来了第三个线程来抢锁，那么轻量级锁就会膨胀成**重量级锁**，其他等待的线程就会进入**阻塞状态**。
### Volatile 与 Synchronized 的区别
- volatile 用于修饰变量， 确保该变量的更新操作 对于所有线程都是可见的，一旦某一个线程修改了volatile变量，其他所有的线程都可以看到最新值
- synchronized用于修饰方法和代码块，实现了互斥访问，确保同一时刻只有一个线程可以执行该方法或代码块
