### 常见的集合框架
#### Collection
- List是 有序、可重复 的集合，如封装了动态数组的ArrayList和封装了链表的LinkedList
- Set是 无需、不可重复的集合，如 HashSet和TreeSet
- Queue是 队列，如双端队列 ArrayDeque 和 优先级队列 PriorityQueue
#### Map
- Map代表键值对集合，如 HashMap

#### 集合框架的常用工具类
- Collections

### List
#### 介绍一下队列
- Java中的队列主要是通过 Queue接口和并发包下的BlockingQueue 两个接口来实现
- 优先级队列PriorityQueue 是一个无界队列，内部元素 按照 自然排序 或 Comparator 比较器进行排序
- 双端队列 **ArrayDeque** 是一个基于数组的，可以在两端插入和删除元素的队列。
- LinkedList 实现了 Queue 接口的子类 Deque，所以也可以当做双端队列来使用。
#### 用过的一些集合类
- ArrayList: 动态数组，实现了List接口，支持动态增长，
- LinkedList：双向链表，实现了List接口，支持快速插入和删除操作
- HashMap：基于哈希表的Map实现，存储键值对，通过键来快速查找
- HashSet：基于HashMap的Set集合实现，用来存储唯一元素
- TreeMap：基于红黑树的有序Map集合，可以按键的顺序排序
- PriorityQueue：优先级队列，可以按照比较器或者元素的自然顺序排序

#### ArrayList 和 LinkedList区别
- ArrayList是基于数组实现的，LinkedList是基于链表实现的
- ArrayList 更偏好于元素的查找，LinkedList更偏向于元素的增删
- ArrayList 在数组中间位置进行元素的增删，会需要移动大量的元素，因此时间复杂度会变成O(n)，而LinkedList进行元素的增删只需要关注于前置结点、后置结点和插入节点
- ArrayList可以支持随机访问，但LinkedList不支持随机访问
- ArrayList 是基于数组的，是一块连续的内存空间，所以它的内存占用是比较紧凑的；但如果涉及到扩容，就会重新分配内存，空间是原来的 1.5 倍；LinkedList 是基于链表的，每个节点都有一个指向下一个节点和上一个节点的引用，于是每个节点占用的内存空间比 ArrayList 稍微大一点。
#### ArrayList 和 LinkedList 使用环境
- ArrayList 适用于
	- 随机访问频繁
	- 读取操作远多于写入操作
	- 在末尾增删元素
- LinkedList适用于
	- 顺序访问频繁
	- 在列表中频繁插入删除元素的场景
	- LinkedList可以实现队列FIFO和栈LIFO

#### 数组和链表什么区别
- 数组在内存中需要占用一块连续的存储空间。数组在创建的时候就确定了大小，一旦分配内存就不能改变了
- 链表的元素存储在于内存中的任意位置，每个节点通过指针指向下一个节点。

#### 为什么ArrayList不直接序列化数组元素？
- ArrayList 确实实现了 Serializable 接口 。但ArrayList的底层ElementData是加了 transient 关键字。 这是因为ArrayList 往往会有很多空闲容量，如果直接序列化会严重浪费内存
- ArrayList 重写了方法 writeObject和readObject，根据ArrayList 的 size 来循环序列化每一个元素


### HashMap
#### HashMap的底层结构
- JDK 8 中的HashMap 的数据结构是数组＋链表+红黑树
- 数组用来存储键值对，每个键值对可以通过索引直接拿到，索引是对 键 的哈希值进一步hash（）得到的
- 当多个键经过哈希处理得到相同的索引后，就通过链表来解决哈希冲突
- 当链表长度超过8且数组长度大于64时，链表就会转换成红黑树
- HashMap 的初始容量是 16，随着元素的不断添加，HashMap 就需要进行扩容，阈值是`capacity * loadFactor`，capacity 为容量，loadFactor 为负载因子，默认为 0.75。
#### 负载因子
- 用来衡量哈希表的填充程度，是一个0到1之间的数值
- 负载因子过高-接近1，会导致哈希冲突增加
- 负载因子过低-接近0，会浪费内存，存在大量未被使用的空间
- 默认负载因子是0.75
#### HashMap 的put流程
- 哈希寻址 → 处理哈希冲突（链表还是红黑树）→ 判断是否需要扩容 → 插入/覆盖节点。
- 第一步是通过哈希方法去扰动哈希值，减少哈希冲突
- 第二步是 先将数组第一次扩容，也就是从null 到 有。使用哈希值和数组长度进行取模运算，确定索引位置
	- 如果当前位置为空，那么就直接插入，如果当前位置存在key，就判断key是否相同，如果相同那就直接覆盖之前的value，否则就发送了哈希冲突
	- 如果当前是链表，则添加新节点到链表尾部。如果长度大于8，就转换成红黑树
- 每次插入后都需要判断是否需要扩容

#### HashMap的扩容机制
- 扩容时，HashMap会创建一个新的数组，容量为之前的两倍，然后遍历原HashMap中的元素，将其填入新的HahsMap中
- 如果当前桶中只有一个元素，那么就直接通过键的哈希值与表长进行取模来得到索引
- 如果当前桶是红黑树，那么通过split方法分裂节点从而使得树平衡
- 如果当前桶是链表，那么将原哈希值和原数组长度来做与运算，并根据此来判断索引是否要移动，如果要移动，就移动原数组长度
- 在JDK 7 的扩容中，使用头插法来重新插入原节点，这会导致链表无法保证原有的顺序
- 在JDK 8 中，修改成了尾插法并且当 原哈希值和原长度进行与运算为0 时，元素保留在原索引的位置；否则元素移动到原索引 + 旧数组大小的位置。
- 因此扩容的时候不需要对每一个节点进行位运算，关注于 原哈希值和原长度 就行了
#### HashMap 为什么 不是线程安全的
- 多线程下扩容会造成死循环，在JDK7中的HashMap使用头插法去处理链表，多线程环境下容易造成死循环；JDK8解决了这个问题
- 多线程在put元素时，可能会导致元素丢失。两个线程都要往同一个位置放入元素，可能会发生覆盖
- put 和 get 并发时，可能导致 get 为 null。put在进行扩容的for循环，但是for循环还没结束就进行get操作
#### 怎么解决HashMap 线程不安全 
- 使用加了synchronized关键字的HashTable；使用方法返回一个线程安全的Map；使用并发工具包的ConcurrentHashMap
- 在早期JDK 中，使用HashTable 来保证线程安全，HashTable在方法上添加了synchronized 关键字
- 也可以通过Collections.synchronizedMap去返回一个线程安全的HashMap，内部是通过 synchronized 对象锁来保证线程安全的
- 更优雅的解决方案是使用并发工具包下的 [ConcurrentHashMap](https://javabetter.cn/thread/ConcurrentHashMap.html)，使用了[CAS](https://javabetter.cn/thread/cas.html)+ [synchronized 关键字](https://javabetter.cn/thread/synchronized-1.html)来保证线程安全。
### ConcurrentHashMap 的实现
- JDK7 中，采用分段锁，整个Map 会被分为 若干段，每个段都可以独立加锁，不同线程之间操作不同的锁，实现并发
- JDK8 中，采用了更加细粒度的桶锁，配合CAS和synchronized 代码块 控制并发写入，最大程度减少锁的竞争
- 对于读操作，ConcurrentHashMap 使用了volatile变量来保证了内存的可见性
- 对于写操作，ConcurrentHashMap 优先使用 CAS尝试插入，如果成功就直接返回；否则用Synchronized 代码块进行加锁处理
### JDK7中的ConcurrentHashMap
-  JDK7 中，采用分段锁，整个Map 会被分为 若干段，每个段都可以独立加锁，每个端类似一个Hashtable
- 每个段维护一个键值对数组，HashEntry是一个单向链表
- 段继承了ReentrantLock，所以每个段都是一个可重入锁，
