# 个人项目架构大重构：从“手动拖拽”到“自动化魔法”✨

## 引言：告别刀耕火种 🔥

每个开发者都曾经经历过那段**“手动拖拽”的黑暗时期**。你打开终端，输入 `npm run build`，然后心情沉重地登录到远程服务器，手动拖拽文件、替换文件夹，等着这些过程完成。每一次修改错别字，都要重复这一系列繁琐的操作，这不仅耗时，还让人心力交瘁。更糟糕的是，这样的手动流程充满了出错的风险：拖错了文件，甚至可能把重要内容替换掉，几乎成为一种“技术性赌博”🎰。

但我决定迎接挑战，突破这些局限。这篇文章将带你一起走过我如何将我的个人知识库从“本地构建 + 手动拖拽 FTP/SSH 上传”的原始时代，爆改成了现代化的 CI/CD 工作流的过程。🌱

### 我的终极愿景是什么？  
**写完笔记，只需一次 `git push`，剩下的交给魔法！** 🎩✨

没错，自动化部署是我的目标。我不再需要亲自拖拽文件，每次修改之后，只要 `git push`，系统会自动执行构建、测试和部署，整个过程流畅且高效，完美实现了“基础设施即代码”的梦想！

## 全局架构设计 🌍

让我们一步步分析这套现代化 CI/CD 流程是如何运作的。基于 **Obsidian** 写作，**GitHub** 托管，**GitHub Actions** 自动化流水线，和 **Docker + Nginx** 容器化部署，我构建了这一高效系统。

### 工作流简述：
1. **Obsidian 本地写作**：在本地使用 Obsidian 完成笔记和内容的创建。Obsidian 的链接和层级结构非常适合我的知识库管理需求。
2. **GitHub 托管**：所有笔记和项目文件同步至 GitHub 仓库，以便进行版本控制。
3. **GitHub Actions 自动化流水线**：每当我提交笔记后，通过 Webhook 触发 GitHub Actions 执行自动化构建和部署。
4. **Docker + Nginx 容器化部署**：GitHub Actions 自动登录服务器，拉取最新代码、重新构建 Docker 镜像，并通过 Nginx 部署新的应用。

整个过程如同魔法般顺利，一切都自动完成。🔥

## 炼狱踩坑录：连斩四大技术 BOSS ⚔️

在重构过程中，我遇到了几个技术难题，几乎让我想要放弃。但正是这些挑战，塑造了我的**工程师思维**，让我不断迭代和优化我的系统。接下来，让我带你一起走进这些跌宕起伏的**打怪升级史诗**！

### ⚔️ BOSS 1：权限门卫的无情拒绝 (SSH 认证与 Linux 权限) 🔐

**表现：** 每次 GitHub Actions 尝试使用 `appleboy/ssh-action` 登录服务器时，总是报错，连接总是失败。

**原理与破局：**  
GitHub Actions 通过 SSH 密钥来连接服务器，但问题不仅仅在密钥的配置。Linux 的安全机制极其严格，密钥文件的权限必须正确设置，否则 **sshd** 会拒绝连接。

- `~/.ssh` 目录权限必须是 `700`（只有属主可读、写、执行）。
- `authorized_keys` 文件权限必须设置为 `600`（只有属主可读、写）。

如果权限设置不对，密钥就会被认为不安全，从而无法登录。解决方案是：确保这两个文件的权限都设置正确。

### ⚔️ BOSS 2：被 232MB 巨兽拖垮的流水线 (构建上下文溢出) 🐉

**表现：** 在执行 `docker build` 时，Docker 报告 `Sending build context to Docker daemon 232.6MB`，然后发生 Timeout 崩溃。

**原理与破局：**  
这是一个典型的“构建上下文溢出”问题。因为 Docker 会将整个项目的文件（包括大量不必要的文件，如 `node_modules`）传送到 Docker 守护进程中。这个传输过程非常耗时，尤其是当你没有正确管理 `.gitignore` 时，`node_modules` 文件夹会被提交到 Git 仓库，导致 Docker 每次都要扫描这些无用文件。

**解决方法：**
1. **删除不必要的文件：** 使用 `git rm -r --cached node_modules` 移除 `node_modules` 文件夹。
2. **优化 `.gitignore`：** 确保 `.gitignore` 文件里有正确的排除规则。
3. **清理服务器：** 在服务器端执行 `rm -rf node_modules` 清理无用文件，极大提升构建速度。

### ⚔️ BOSS 3：跨洋网络的阻击 (Alpine 源与 NPM 镜像超时) 🌏

**表现：** 执行 `apk add git` 和 `npm install` 时，Docker 内的网络连接经常中断，提示 `network connection aborted`。

**原理与破局：**  
Docker 默认使用的 Alpine Linux 镜像源和 NPM 源位于海外，导致国内服务器访问这些源时非常不稳定。为了解决这一问题，我们需要**更换镜像源**，使用国内的镜像源。

**解决方法：**
1. **替换 Alpine 镜像源：** 使用 `sed` 命令将 `dl-cdn.alpinelinux.org` 替换为 `mirrors.aliyun.com`。
2. **更换 NPM 镜像源：** 切换到腾讯云的 NPM 镜像源，使用 `npm config set registry https://mirrors.cloud.tencent.com/npm/`。

这一步骤让我从“等到天荒地老”到“3秒钟搞定”！⏱️

### ⚔️ BOSS 4：VitePress 的傲娇路由与“乾坤大挪移” 🔄

**表现：** Rollup 编译第一阶段成功，但在 `rendering pages...` 阶段疯狂报错 `ERR_MODULE_NOT_FOUND`，导致 Docker 始终部署旧的缓存版本，只显示 Nginx 的蓝屏欢迎页。

**原理与破局：**  
VitePress 对于静态资源的路径有着严格的要求。如果你在本地使用的是 `/images/xxx.png` 这种路径，VitePress 会在构建时将其解析为错误的路径，导致页面渲染失败。

**解决方法：**  
我没有选择手动修改几百篇笔记中的图片路径，而是使用了**自动化脚本**。在 Dockerfile 中加入以下命令，自动将本地的 `images` 文件夹复制到构建后的 `public` 目录，确保路径正确：

```bash
RUN mkdir -p my-knowledge-base/docs/public && \
    cp -r my-knowledge-base/images my-knowledge-base/docs/public/
```
通过这种方式，整个流程完美自动化，既保留了本地的路径，又保证了云端的渲染正确。
## 终极武器库 💥
经过几轮迭代，我最终完成了完美的部署脚本——**Dockerfile** 和自动化部署脚本。在这部分，我将展示最终版本的 Dockerfile，并给出一些关键注释，帮助你理解每一行的作用。

## 最终 Dockerfile 配置

### 阶段一：构建阶段

```dockerfile
FROM node:18-alpine AS builder
WORKDIR /app

# 🌟 核心加速 1：替换阿里云镜像源，解决 apk add 超时
RUN sed -i 's/dl-cdn.alpinelinux.org/mirrors.aliyun.com/g' /etc/apk/repositories && \
    apk add --no-cache git

COPY package.json package-lock.json ./ 

# 🌟 核心加速 2：使用国内 NPM 镜像
RUN npm config set registry https://mirrors.cloud.tencent.com/npm/ && npm install

COPY . .

# 🌟 核心黑魔法：打包前，自动将图片搬运至 public 目录，完美兼容 Obsidian 路径！
RUN mkdir -p my-knowledge-base/docs/public && \
    cp -r my-knowledge-base/images my-knowledge-base/docs/public/

RUN npx vitepress build my-knowledge-base/docs
```
### 阶段二：部署阶段
```
FROM nginx:alpine
COPY --from=builder /app/my-knowledge-base/docs/.vitepress/dist /usr/share/nginx/html
EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]
```
