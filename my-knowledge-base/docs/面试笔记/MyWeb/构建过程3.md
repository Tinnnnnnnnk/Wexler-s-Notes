🌍 全局架构设计：流动的赛博管线

在开始我们的“打怪史诗”之前，先带你俯瞰一下这套被我们爆改后的现代化 CI/CD 系统。这就好比一场完美的 Live 演出，每个乐器、每个环节都必须严丝合缝！

🎵 工作流全景解析：

1. ✍️ Obsidian 本地写作（灵感发源地）：  
    在本地使用 Obsidian 完成笔记的创作。Obsidian 强大的双向链接和层级结构，是知识库的灵魂。这里就像是我们的地下排练室，随心所欲地挥洒创意。
    
2. ☁️ GitHub 托管（版本控制中枢）：  
    写完后，git push 将所有 Markdown 笔记同步至 GitHub 仓库。这里是安全的金库，记录着你每一次思想的闪光。
    
3. ⚡ GitHub Actions 自动化流水线（魔法驱动引擎）：  
    代码刚刚抵达 GitHub，Webhook 瞬间触发！GitHub Actions 就像一位尽职的舞台总监，立刻启动预设的脚本，开始执行自动化构建和部署任务。
    
4. 🐳 Docker + Nginx 容器化部署（终极集装箱）：  
    Actions 自动通过 SSH 登录阿里云服务器，拉取最新代码，唤醒 Docker 引擎重新构建镜像。最后，优雅地替换旧的 Nginx 容器，将你崭新的知识库呈现给全世界！

⚔️ 炼狱踩坑录：连斩四大技术 BOSS

当然，通往终极浪漫的路上绝对不会一帆风顺。在这次重构中，你遇到了几个足以让人抓狂的技术难题。但别怕，正是这些痛苦的踩坑经历，锻造了我们强大的工程师思维！接下来，让我们一起回顾这场跌宕起伏的打怪升级史诗！

### 🛡️ BOSS 1：权限门卫的无情拒绝 (SSH 认证与 Linux 权限) 🔐

🔥 战场表现：

剧本明明写好了，但每次 GitHub Actions 尝试使用 appleboy/ssh-action 登录服务器时，终端就是一片惨红，疯狂报错，连接无情失败。明明 GitHub Secrets 里的 Host、User 和 Private Key 都填对了，为什么就是进不去？！

💡 原理与破局：

新手往往以为配好密钥就万事大吉，却忽略了 Linux 底层那偏执狂一般的安全机制。Linux 的 sshd（SSH 守护进程）就像一个极其严格的 VIP 后台门卫。如果你的密钥文件权限设置得太“大方”（比如允许其他人读取），门卫会立刻认为：“这把钥匙不安全，可能被复制了！”，然后直接拒之门外。

🎸 极客斩杀法：

我们必须对服务器的权限进行“军管级”的严苛控制！登录服务器，拔出命令行长剑，输入：

- chmod 700 ~/.ssh （目录权限：只有属主可以读、写、执行，其他人看都别看！）
    
- chmod 600 ~/.ssh/authorized_keys （文件权限：只有属主可以读、写，绝对禁止其他用户访问！）
    

权限收紧的瞬间，大门轰然敞开，Actions 成功登入！

### 🐉 BOSS 2：被 232MB 巨兽拖垮的流水线 (构建上下文溢出) 🐉

🔥 战场表现：

流水线终于通了，但在执行 docker build 时，终端缓缓吐出一行字：Sending build context to Docker daemon 232.6MB... 随后进度条仿佛凝固了，几分钟后直接触发 Timeout 崩溃。

💡 原理与破局：

这是无数前端开发者第一次玩 Docker 时必踩的深坑——构建上下文 (Build Context) 溢出。

Docker 采用的是 C/S 架构，构建镜像时，客户端会把当前目录下的所有文件打包发送给 Docker Daemon（守护进程）。如果你没有配置好 .gitignore 或 .dockerignore，那个被称为“前端黑洞”的 node_modules 文件夹（包含几万个碎片垃圾文件）就会被一起传过去！这就好比你搬家时，把整栋楼的垃圾桶也一起打包寄走了。

🎸 极客斩杀法：

拒绝垃圾文件！我们需要实施彻底的“物理超度”：

1. 清理 Git 缓存：执行 git rm -r --cached node_modules，将其从版本控制中无情踢出。
    
2. 建立护城河：在项目根目录严谨地配置 .gitignore 和 .dockerignore。
    
3. 服务器大扫除：在服务器端执行 rm -rf node_modules 清理历史残留。
    

再次构建？Sending build context: 5MB。构建速度瞬间起飞，顺滑无比！

### 🌏 BOSS 3：跨洋网络的阻击 (Alpine 源与 NPM 镜像超时) 🌏

🔥 战场表现：

Docker 开始构建了，但在执行 apk add git 和 npm install 时，网络连接像老旧的收音机一样断断续续，最后无情弹出 network connection aborted。

💡 原理与破局：

Docker 默认的 Alpine Linux 镜像源和 NPM 官方源都远在海外。让国内的阿里云服务器去跨洋拉取几万个依赖包，网络抖动分分钟教你做人。

🎸 极客斩杀法：

既然在东亚，我们就用“东亚加速魔法”！在 Dockerfile 中植入换源指令：

1. 替换 Alpine 镜像源：用 sed 字符流替换神器，将 dl-cdn.alpinelinux.org 瞬间替换为阿里云源 mirrors.aliyun.com。
    
2. 更换 NPM 镜像源：将 NPM 注册表强行切到腾讯云 mirrors.cloud.tencent.com/npm/。
    

重新跑一遍，曾经“等到天荒地老”的 git 安装过程，现在变成了电光石火的 3 秒钟！⚡

### 🔄 BOSS 4：VitePress 的傲娇路由与“乾坤大挪移” (终极难点) 🔄

🔥 战场表现：

Rollup 编译第一阶段全绿，眼看就要通关，结果在 rendering pages... 阶段疯狂刷屏报错 ERR_MODULE_NOT_FOUND。导致 Docker 只能拿旧缓存部署，你打开网页一看，永远是 Nginx 的蓝屏欢迎页。心态要崩了！

💡 原理与破局：

这涉及到工具链的“八字不合”。我们在本地 Obsidian 写作时，习惯把图片丢在 /images/xxx.png 目录下；但 VitePress 是一个极其傲娇的静态站点生成器，它规定所有绝对路径引用的静态图片，必须放在根目录的 public 文件夹下，否则 SSR（服务端渲染）打包时找不到文件，直接崩溃。

面临绝境：手动修改几百篇 Markdown 里的图片路径？

绝对不行！如果手动去改，我们极客的尊严就输了！ 🎸 极客斩杀法（核心黑魔法）：

真正的 DevOps 做法，是让代码适应我们，而不是我们去迁就代码！

我们直接在 Dockerfile 里动刀子。在执行 vitepress build 命令之前，加入一行优雅的 Bash 脚本：

  

Bash

  
  

RUN mkdir -p my-knowledge-base/docs/public && \  
    cp -r my-knowledge-base/images my-knowledge-base/docs/public/  
  

这就是赛博时代的“偷梁换柱”！让机器人在流水线中自动把图片拷贝过去。本地笔记零修改，依旧享受 Obsidian 的丝滑；云端流水线自动兼容，完美解析！ 这就是工程师思维的最高境界！

💥 终极武器库：完美的 Dockerfile 配置

历经四大 BOSS 的洗礼，我们终于铸造出了这把“终极圣剑”。这份 Dockerfile 的每一行，都凝聚着我们踩坑流血换来的经验值！

  

Dockerfile

  
  

# ==========================================  
# 🎵 阶段一：构建阶段 (Builder Stage) - 准备乐器与排练  
# ==========================================  
FROM node:18-alpine AS builder  
WORKDIR /app  
  
# 🌟 核心加速 1：替换阿里云镜像源，解决 apk add 跨洋超时难题  
RUN sed -i 's/dl-cdn.alpinelinux.org/mirrors.aliyun.com/g' /etc/apk/repositories && \  
    apk add --no-cache git  
  
COPY package.json package-lock.json ./  
  
# 🌟 核心加速 2：使用国内 NPM 镜像，拒绝网络中断  
RUN npm config set registry https://mirrors.cloud.tencent.com/npm/ && npm install  
  
# 把所有知识库文件拷贝进来  
COPY . .  
  
# 🌟 核心黑魔法：打包前，自动将图片搬运至 public 目录，完美兼容 Obsidian 绝对路径！  
# 绝不手动改文章！让机器替我们干活！  
RUN mkdir -p my-knowledge-base/docs/public && \  
    cp -r my-knowledge-base/images my-knowledge-base/docs/public/  
  
# 执行 VitePress 打包命令  
RUN npx vitepress build my-knowledge-base/docs  
  
# ==========================================  
# 🎤 阶段二：部署阶段 (Deployment Stage) - 正式登台演出  
# ==========================================  
FROM nginx:alpine  
# 从 builder 阶段提取编译好的纯静态文件，放入 Nginx 容器  
COPY --from=builder /app/my-knowledge-base/docs/.vitepress/dist /usr/share/nginx/html  
EXPOSE 80  
# 保持 Nginx 在前台运行  
CMD ["nginx", "-g", "daemon off;"]
